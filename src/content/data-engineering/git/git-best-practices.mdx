## Buenas prácticas y .gitignore

Cierras este módulo con los hábitos que te separan del usuario casual. Saber
los comandos es 30% del trabajo; el otro 70% es disciplina al usarlos: qué
mensaje pones, qué archivos ignoras, cuándo ramificas. Esta lección recoge las
prácticas más rentables — las que tu yo del futuro y los revisores de tus PRs
te van a agradecer.

### Mensajes de commit que valen la pena

Un commit es una pequeña unidad de cambio + un mensaje. El **mensaje es para
humanos**: el `git log` que tu equipo va a leer dentro de 6 meses cuando algo
se rompa.

**Reglas mínimas:**

1. **Primera línea ≤ 72 caracteres.** El subject. Lo que ves en `git log
   --oneline`. Imperativo, en presente: "agrega" no "agregando" ni "agregó".
2. **Línea en blanco** entre subject y cuerpo (si hay cuerpo).
3. **Cuerpo opcional** explicando el porqué cuando no es obvio. Las líneas a
   ~72 caracteres también.
4. **Prefijo semántico** (Conventional Commits): `feat`, `fix`, `docs`,
   `refactor`, `test`, `chore`, `perf`, `build`, `ci`.

Ejemplo completo:

```
fix(pipeline): handle null amount in payments aggregation

Stripe webhook can send amount=null when a charge is voided
mid-processing. Previously this crashed the daily aggregation
with TypeError. Now we coalesce null to 0 and log a warning.

Refs: DATA-142
```

¿Por qué importa? Porque dentro de 6 meses, cuando estés depurando un bug,
vas a hacer:

```bash
git log --oneline --grep="payment"
# fix(pipeline): handle null amount in payments aggregation
# feat(pipeline): add payments aggregation
# ...
```

Y verás directamente qué buscar. Sin convención, ese log es ilegible.

### Commits atómicos

Un commit debe representar **una idea coherente**. Si tu mensaje tiene "y",
es señal de que son dos commits distintos.

Mal:

```
fix bug + add feature + update README
```

Bien (3 commits):

```
fix(api): retry on 5xx with exponential backoff
feat(metrics): add request latency histogram
docs: update setup instructions for new metrics
```

Beneficios:
- `git revert` puede deshacer un cambio sin tocar los otros.
- `git bisect` (debugging por bisección) localiza qué commit introdujo un bug.
- El review es focalizado.

### .gitignore: qué nunca debe entrar al repo

`.gitignore` es un archivo (uno o más, pueden anidarse en subcarpetas) donde
listas patrones que Git debe **ignorar completamente**. Archivos que cumplan
esos patrones no aparecen en `git status` ni los puedes `add` por accidente.

Categorías de archivos a ignorar:

**1. Generados / builds:**
```
dist/
build/
out/
*.pyc
__pycache__/
node_modules/
.next/
target/
```

**2. Archivos de IDE / sistema:**
```
.vscode/        # solo si NO compartes config; algunos equipos sí versionan
.idea/
.DS_Store       # macOS
Thumbs.db       # Windows
```

**3. Logs y datos temporales:**
```
*.log
logs/
*.tmp
.cache/
```

**4. Datos brutos / outputs (Data Engineering):**
```
data/raw/
data/processed/
*.parquet
*.csv
*.xlsx
```

**5. Secretos y configuración local:**
```
.env
.env.local
*.pem
*.key
secrets.json
config.local.yaml
```

Los secretos son **lo más crítico**. Una clave de AWS commiteada a un repo
público se descubre en menos de 5 minutos por bots que vigilan GitHub. Si
pasa, considera la clave comprometida y rótala inmediatamente — no basta con
borrar el commit.

### Plantillas de .gitignore

GitHub mantiene plantillas oficiales por lenguaje:

```bash
# En el repo:
curl https://raw.githubusercontent.com/github/gitignore/main/Python.gitignore > .gitignore
```

Existen para Python, Node, Java, Go, etc. Empieza por una plantilla y agrega
lo específico de tu proyecto.

### Si ya commiteaste algo que no debías

Caso común: pusheaste un `.env` con una clave. Pasos:

1. **Rota la credencial inmediatamente** en el provider (AWS, Stripe, etc).
   Asume que ya está comprometida.
2. Agrega `.env` al `.gitignore`.
3. Quita el archivo del repo (sin borrarlo localmente):
   ```bash
   git rm --cached .env
   git commit -m "chore: remove .env from tracking"
   git push
   ```
4. Si quieres borrarlo del **historial completo** (no solo del último commit),
   usa `git filter-repo` o `BFG Repo Cleaner`. Es más invasivo y reescribe
   historia — coordínalo con el equipo.

Aun después de borrarlo del histórico, considera la clave comprometida. El
paso 1 es no negociable.

### Otras buenas prácticas

- **Nombra ramas con prefijo y kebab-case**: `feature/null-handling`,
  `fix/api-timeout`, `chore/upgrade-deps`. Algunos equipos usan
  `tu-usuario/feature-x` para identificar autoría.
- **Rebasea ramas largas frecuentemente** sobre `main` para evitar conflictos
  acumulados:
  ```bash
  git switch feature/long-running
  git fetch origin
  git rebase origin/main
  ```
- **Una vida laboral entera con `git pull --rebase` por defecto**:
  ```bash
  git config --global pull.rebase true
  ```
  Mantiene el histórico lineal en lugar de generar merge commits cada vez que
  haces pull.
- **No commits con WIP en `main`**. Usa una rama o, si tienes que pausar
  rápido, `git stash`.
- **Activa signed commits** si tu organización lo requiere (firma con GPG o
  SSH). Probacaron que el commit fue autorizado por ti.

### Errores comunes

- **Repos con `data/`, `node_modules/` y `__pycache__/` versionados**. El repo
  pesa 2GB, los pulls son lentos, los diffs son ilegibles.
- **Commits con mensaje "wip" o "fix"**. No transmiten información. Equivalen
  a no tener mensaje.
- **`force-push` a `main`**. Borra trabajo de otros sin avisar. Aunque sea
  "tu" repo, no lo hagas — vas a olvidar y luego pasa.
- **Mantener una rama de 6 meses sin rebasear**. Mergearla es un infierno de
  conflictos. Ramas cortas, siempre.
- **Borrar `.git/`**. Pierdes todo el histórico. No hay vuelta atrás. Sí,
  pasa más de lo que crees.

  language="javascript"
  starterCode={`function shouldIgnore(path, patterns) {
  // path: 'src/__pycache__/foo.pyc'
  // patterns: ['__pycache__/', '*.log', '.env']
  return false;
}

console.log(shouldIgnore('src/__pycache__/foo.pyc', ['__pycache__/']));
console.log(shouldIgnore('app.log', ['*.log']));
console.log(shouldIgnore('app.py', ['*.log']));`}
  solution={`function shouldIgnore(path, patterns) {
  for (const p of patterns) {
    if (p.endsWith('/')) {
      const dirName = p.slice(0, -1);
      if (path.split('/').includes(dirName)) return true;
    } else if (p.startsWith('*.')) {
      const ext = p.slice(1);
      if (path.endsWith(ext)) return true;
    } else {
      if (path === p || path.endsWith('/' + p)) return true;
    }
  }
  return false;
}
console.log(shouldIgnore('src/__pycache__/foo.pyc', ['__pycache__/']));
console.log(shouldIgnore('app.log', ['*.log']));
console.log(shouldIgnore('app.py', ['*.log']));`}
  hints={[
    'Para patrones que terminan en /, busca si alguna parte del path es ese directorio.',
    'Para *.ext, basta con path.endsWith(".ext").',
    'Para nombres exactos como .env, compara o verifica que termine en /.env.',
  ]}
  tests={[
