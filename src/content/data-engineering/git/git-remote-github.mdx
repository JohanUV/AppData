## Trabajando con GitHub remoto

Hasta aquí Git ha vivido solo en tu máquina. Eso ya es valioso, pero el verdadero
poder llega cuando lo conectas con un **remoto**: un repositorio Git hospedado
en otro lugar (GitHub, GitLab, Bitbucket, o tu propio servidor). El remoto es
el punto de sincronización entre desarrolladores y el archivo seguro de tu
código si tu disco muere.

En esta lección vinculas tu repo local a GitHub, aprendes a sincronizar con
`push`, `pull` y `fetch`, y configuras autenticación con SSH para que no te
pida contraseña en cada operación.

### Crear un repo en GitHub

1. Ve a [github.com](https://github.com), entra a tu cuenta.
2. Esquina superior derecha → **+** → **New repository**.
3. Pon un nombre (`mi-primer-pipeline`), descripción opcional, decide si público
   o privado.
4. **No marques** "Add a README file" — ya tienes uno local. Si lo marcas
   creas un commit en remoto que vas a tener que mergear.
5. Crea el repo.

GitHub te muestra una pantalla con instrucciones. Las que importan son las del
caso "push an existing repository":

```bash
git remote add origin git@github.com:tu-usuario/mi-primer-pipeline.git
git branch -M main
git push -u origin main
```

Antes de ejecutarlas, configuremos SSH.

### Autenticación: HTTPS vs SSH

GitHub acepta dos protocolos:

- **HTTPS** — `https://github.com/usuario/repo.git`. Te pide token cada push
  (las contraseñas ya no se aceptan desde 2021). Sencillo si lo combinas con
  un credential helper.
- **SSH** — `git@github.com:usuario/repo.git`. Una vez configurado, no te pide
  nada. Más cómodo a la larga.

Vamos con SSH (es lo que verás en la mayoría de tutoriales serios).

### Configurar SSH paso a paso

**1. Genera una clave** (si no tienes una):

```bash
ssh-keygen -t ed25519 -C "tu@email.com"
```

Pulsa Enter para aceptar la ruta por defecto (`~/.ssh/id_ed25519`). Pon una
passphrase si quieres seguridad extra (recomendado en máquinas compartidas).

**2. Inicia el agente SSH y agrega la clave:**

```bash
# Linux/macOS
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# Windows con Git Bash
eval $(ssh-agent -s)
ssh-add ~/.ssh/id_ed25519
```

**3. Copia la clave pública:**

```bash
# Linux
cat ~/.ssh/id_ed25519.pub

# macOS (copia al portapapeles)
pbcopy < ~/.ssh/id_ed25519.pub

# Windows Git Bash
cat ~/.ssh/id_ed25519.pub | clip
```

**4. Pégala en GitHub:**

GitHub → Settings → SSH and GPG keys → **New SSH key** → Pega → Save.

**5. Verifica:**

```bash
ssh -T git@github.com
# Hi tu-usuario! You've successfully authenticated...
```

Si ves ese mensaje, todo bien. Ya puedes usar URLs SSH.

### Conectar local con remoto

```bash
# Asociar el remoto (lo llamamos 'origin' por convención)
git remote add origin git@github.com:tu-usuario/mi-primer-pipeline.git

# Ver remotos configurados
git remote -v
# origin  git@github.com:tu-usuario/mi-primer-pipeline.git (fetch)
# origin  git@github.com:tu-usuario/mi-primer-pipeline.git (push)

# Renombrar la rama actual a main si estuviera en master
git branch -M main

# Primer push: -u establece tracking
git push -u origin main
```

`-u` (`--set-upstream`) hace que tu rama local recuerde a qué rama remota está
asociada. Después puedes ejecutar `git push` y `git pull` sin argumentos —
Git sabe a dónde.

### El flujo típico de sincronización

```
mañana:    git pull origin main          # Empiezo el día actualizado
mediodía:  git push origin feature/x     # Subo trabajo de la mañana
tarde:     git fetch && git log ...      # Veo qué hicieron los demás
```

`git fetch` antes de mergear es buen hábito: te muestra qué hay nuevo sin
forzarte a integrarlo.

### Clonar un repo existente

Si te incorporas a un proyecto ya en GitHub:

```bash
git clone git@github.com:org/proyecto.git
cd proyecto
git remote -v       # ya viene configurado
git branch -a       # ramas locales y remotas
```

`git clone` hace tres cosas a la vez: crea la carpeta, descarga todo el
historial, y configura `origin` apuntando al repo de origen.

### Push rechazado: cómo arreglar

Caso típico: pusheaste, te lo rechazó:

```
! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'github.com:user/repo.git'
hint: Updates were rejected because the remote contains work that you do not
hint: have locally.
```

Significa: alguien (incluso tú desde otro equipo) pusheó algo nuevo a `main`
que no tienes. Solución:

```bash
git pull --rebase     # Trae los nuevos commits y reaplica los tuyos encima
git push              # Ahora sí
```

`--rebase` mantiene el histórico lineal (sin merge commits "Merge branch
main"). Es buena práctica para ramas en las que solo trabajas tú.

### Errores comunes

- **Hacer `git push --force` para "arreglar"**. Sobreescribe el remoto y
  destruye trabajo de otros. Solo en ramas personales y nunca a `main`.
- **Olvidar `-u` en el primer push** y luego tener que escribir `origin main`
  en cada push. No es grave pero molesto.
- **Confundir `fetch` con `pull`**. Fetch solo descarga; pull descarga e
  integra. Si no estás seguro de lo que hay en remoto, usa fetch primero.
- **Pushear secretos**. Una vez en GitHub, considera la clave comprometida
  (los crawlers indexan repos públicos en minutos). Rótala y usa `.gitignore`
  desde el inicio.
- **Trabajar con HTTPS y meter el password mil veces**. O configura un
  credential helper, o pásate a SSH.
