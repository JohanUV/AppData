## Instalación y configuración inicial

Antes de hacer commits hay que tener Git instalado y decirle quién eres. Suena
obvio, pero es donde el 80% de los principiantes se traban: instalan Git, hacen
un commit, y aparece un autor llamado `unknown@DESKTOP-XYZ` en el histórico.
Eso luego no se borra fácil. Mejor configurar bien desde el inicio.

En esta lección vamos a:

1. Instalar Git en tu sistema operativo.
2. Configurar tu identidad (nombre, email).
3. Elegir editor por defecto.
4. Verificar que todo funciona con `git --version`.

### Instalación por sistema operativo

**Windows.** Descarga el instalador oficial desde
[git-scm.com/download/win](https://git-scm.com/download/win). Es un `.exe` que
incluye Git Bash, una mini-terminal estilo Linux que necesitarás (PowerShell
también funciona, pero Git Bash es más cercano al ecosistema Unix donde nació
Git). Durante la instalación deja casi todo en defaults, con dos detalles:

- **Default editor**: elige uno que conozcas. VS Code es buena opción si ya lo
  usas.
- **Line endings**: marca "Checkout Windows-style, commit Unix-style". Git
  normalizará los saltos de línea para que tus repos no se llenen de diffs
  fantasma cuando colabores con gente de macOS/Linux.

**macOS.** Tres caminos:

```bash
# Opción A: viene con las Xcode Command Line Tools
xcode-select --install

# Opción B: Homebrew (más actualizado)
brew install git

# Opción C: instalador oficial desde git-scm.com
```

**Linux.** Usa el gestor de paquetes:

```bash
# Debian / Ubuntu
sudo apt update && sudo apt install git

# Fedora / RHEL
sudo dnf install git

# Arch
sudo pacman -S git
```

### Verificar la instalación

Abre tu terminal (Git Bash en Windows, Terminal en macOS, cualquier emulador en
Linux) y ejecuta:

```bash
git --version
```

Deberías ver algo como `git version 2.45.2`. Si tienes 2.30 o superior, todo
bien. Si te aparece "command not found", reinicia la terminal (las variables
PATH no siempre se recargan automáticamente). Si persiste, revisa que la
instalación añadió Git al PATH.

### Configura tu identidad

Cada commit lleva un nombre y un email. Estos no se autentican (Git confía en
ti localmente), pero quedan grabados en el histórico para siempre. Configura
los globales una sola vez:

### Configuraciones útiles adicionales

Estas no son obligatorias pero te van a ahorrar dolor:

```bash
# Mejor formato de log a colores
git config --global color.ui auto

# Aliases que muchos usan
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.lg "log --oneline --graph --all"

# Resolver conflictos con tu editor
git config --global merge.tool vscode
```

Con eso, `git st` equivale a `git status`, y `git lg` te muestra el historial
visual del repo en una línea por commit.

### Dónde se guarda la config

Git tiene **tres niveles** de configuración, en orden de precedencia:

1. **System** (`/etc/gitconfig`) — para todos los usuarios. Rara vez se toca.
2. **Global** (`~/.gitconfig`) — para tu usuario. Aquí va tu identidad.
3. **Local** (`.git/config` dentro del repo) — para un proyecto específico.

Si tu trabajo y tu cuenta personal usan emails distintos, puedes sobreescribir
en cada repo:

```bash
cd ~/proyectos/del-trabajo
git config user.email "tu@empresa.com"
```

Eso solo afecta a ese repo, dejando el global intacto para los demás.

### Errores comunes

- **Olvidar `--global`** y luego preguntarse por qué cada repo te pide el nombre
  otra vez. Si estás configurando tu identidad personal, casi siempre quieres
  `--global`.
- **Email distinto al de GitHub**: tus commits no aparecerán enlazados a tu
  perfil. Siempre el mismo email verificado.
- **Notepad como editor por defecto en Windows**: mata mensajes de commit
  multilínea con encoding raro. Mejor `code --wait`, `nano` o `vim`.
- **Saltarse la configuración de line endings en Windows** y luego pelearse con
  diffs invisibles. `core.autocrlf=true` es el comportamiento correcto.

## Ejercicio práctico

<Exercise
  id="git-config-exercise"
  title="Constructor de comandos config"
  description="Implementa una función que construya el comando git config a partir de una clave y un valor, agregando --global si la clave es de identidad (user.name, user.email)."
  language="javascript"
  starterCode={`function buildConfigCmd(key, value) {
  // Devuelve un string como: git config --global user.email "x@y.com"
  // Solo agrega --global si key empieza por "user."
  return '';
}

const a = buildConfigCmd('user.email', 'me@x.com');
const b = buildConfigCmd('color.ui', 'auto');
console.log(a);
console.log(b);`}
  solution={`function buildConfigCmd(key, value) {
  const flag = key.startsWith('user.') ? '--global ' : '';
  return \`git config \${flag}\${key} "\${value}"\`;
}
const a = buildConfigCmd('user.email', 'me@x.com');
const b = buildConfigCmd('color.ui', 'auto');
console.log(a);
console.log(b);`}
  hints={[
    'String.prototype.startsWith te dice si la key empieza por "user.".',
    'Usa template literals para construir el string final.',
  ]}
  tests={[
    {
      description: 'user.email lleva --global',
      assert: 'buildConfigCmd("user.email","x@y.com") === \'git config --global user.email "x@y.com"\'',
    },
    {
      description: 'color.ui no lleva --global',
      assert: 'buildConfigCmd("color.ui","auto") === \'git config color.ui "auto"\'',
    },
  ]}
