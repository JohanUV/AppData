## Branches y merge

Hasta ahora trabajamos en una sola línea de tiempo: cada commit encima del
anterior. Funciona para un solo desarrollador en un solo problema. Pero la
realidad es otra: estás arreglando un bug urgente cuando llega un ticket nuevo,
o quieres probar una refactorización sin romper la versión que funciona, o tres
personas tocan el mismo repo a la vez.

La respuesta de Git es **branching**: crear líneas de tiempo paralelas que
luego se fusionan. Y aquí Git brilla — branches son baratas, instantáneas y se
combinan con `merge`.

### ¿Qué es una branch realmente?

Una branch en Git **no es una copia del código**. Es solo un puntero móvil a
un commit. Crear una branch es crear un puntero nuevo (operación que toma
microsegundos). Cuando haces commits en esa branch, el puntero avanza.

<Diagram type="mermaid">
{`gitGraph
  commit id: "init"
  commit id: "feat A"
  branch feature/x
  commit id: "wip 1"
  commit id: "wip 2"
  checkout main
  commit id: "feat B"`}
</Diagram>

`main` y `feature/x` son simplemente etiquetas que apuntan a commits distintos.
Tienen el mismo histórico hasta el punto donde divergieron.

### Operaciones básicas

```bash
# Listar ramas (la actual lleva *)
git branch

# Crear una rama (no cambia a ella todavía)
git branch feature/limpieza-nulls

# Crear y cambiar en un solo paso (lo más común)
git switch -c feature/limpieza-nulls
# o el clásico:
git checkout -b feature/limpieza-nulls

# Cambiar a una rama existente
git switch main
git checkout main

# Borrar una rama (debe estar fusionada)
git branch -d feature/limpieza-nulls

# Borrar a la fuerza (incluso sin merge)
git branch -D feature/limpieza-nulls
```

`git switch` es el comando moderno (Git ≥ 2.23) específico para cambiar de
rama. `git checkout` también funciona y verás ambos en tutoriales — son
intercambiables para esto.

### Trabajar en una rama

Flujo típico:

```bash
# Empiezo desde main actualizado
git switch main

# Creo rama temática
git switch -c feature/handle-nulls

# Edito, agrego, commiteo varias veces
echo "# fix nulls" >> pipeline.py
git add pipeline.py
git commit -m "feat: skip null amounts"

# Más cambios...
git commit -am "test: add null case"
```

Tienes 2 commits nuevos en `feature/handle-nulls`. `main` sigue en su sitio.
Si te das cuenta de que el cambio era una mala idea: te cambias a `main` y
borras la rama, listo, no contaminaste nada.

### Merge: combinar ramas

Cuando termines tu feature, quieres llevar esos commits de vuelta a `main`:

```bash
git switch main
git merge feature/handle-nulls
```

Hay tres escenarios posibles:

**1. Fast-forward.** Si `main` no avanzó desde que creaste la rama, Git
simplemente mueve el puntero `main` al último commit de `feature/handle-nulls`.
No hay merge commit, el histórico queda lineal.

```bash
git merge feature/handle-nulls
# Updating e8a91b2..3f12c4a
# Fast-forward
```

**2. Three-way merge.** Si `main` avanzó por su lado mientras tú trabajabas
en la rama, Git crea un **merge commit** que tiene dos padres: el último de
main y el último de tu rama. El historial pasa a tener forma de Y.

<Diagram type="mermaid">
{`gitGraph
  commit id: "init"
  branch feature
  commit id: "f1"
  commit id: "f2"
  checkout main
  commit id: "m1"
  merge feature id: "merge"`}
</Diagram>

**3. Conflicto.** Si las dos ramas modificaron las mismas líneas del mismo
archivo de forma incompatible, Git no sabe cuál ganar y te pide ayuda.

### Resolver conflictos

Cuando hay conflicto, Git pausa el merge y marca los archivos:

```python
def parse_amount(s):
<<<<<<< HEAD
    return float(s)
=======
    return float(s.strip()) if s else 0
>>>>>>> feature/handle-nulls
```

- Entre `<<<<<<< HEAD` y `=======` está la versión de tu rama actual (donde
  estabas al hacer el merge).
- Entre `=======` y `>>>>>>>` está la versión de la rama que mergeas.

Tú decides la versión final. Editas el archivo dejándolo como debería quedar
(borrando los marcadores), por ejemplo:

```python
def parse_amount(s):
    return float(s.strip()) if s else 0
```

Después marcas el conflicto resuelto y completas el merge:

```bash
git add pipeline.py
git commit       # Git ya tiene un mensaje de merge precargado
```

Si te arrepentiste a mitad:

```bash
git merge --abort
```

Vuelves al estado anterior al `git merge`, sin cambios.

### Estrategia: ramas cortas y temáticas

Una práctica común y muy buena:

- **Una rama por feature/bug.** Nombres descriptivos:
  `feature/null-handling`, `fix/api-timeout`, `chore/update-deps`.
- **Vida corta**: ideal 1-3 días. Mientras más tiempo viva, más difícil será
  mergear (más cambios divergentes en main).
- **Mergeas pronto**, mergeas seguido. La regla: si terminaste algo coherente,
  mergea. No acumules.

Así se ve un repo sano:

```
main:    A---B---C---D---E---F   (siempre verde, siempre deployable)
              \\         \\
              feat/x    feat/y   (ramas cortas, mergeadas pronto)
```

Y así uno enfermo:

```
main:    A---B---C---D
              \\
              feat/refactor-todo   (vida 2 meses, 80 commits, imposible mergear)
```

### Errores comunes

- **Trabajar directamente en main**. Te pierdes el beneficio del aislamiento.
  Disciplínate: rama nueva para cada cosa.
- **Crear ramas sin actualizar main primero**. Bases tu trabajo en código
  viejo. Empieza con `git switch main && git pull` (lo verás en lección 6).
- **Borrar una rama no mergeada con `-D`** y perder el trabajo. Si tenías
  cambios importantes ahí, recuperar requiere `git reflog` (avanzado).
- **Resolver conflictos sin entender qué pasó**. Borrar los marcadores `<<<<`
  no es resolver: hay que pensar qué versión es correcta.
- **Hacer un merge enorme tras semanas de divergencia**. Conflictos múltiples
  en cascada. Por eso ramas cortas.

  starterCode={`function canFastForward(target, source) {
  // target y source son arrays de commits desde el más viejo al más nuevo.
  // Devuelve true si target es prefijo de source.

  solution={`function canFastForward(target, source) {
  if (target.length > source.length) return false;
  for (let i = 0; i < target.length; i++) {
    if (target[i] !== source[i]) return false;
  }
  return true;
}
console.log(canFastForward(['A','B'], ['A','B','C','D']));
console.log(canFastForward(['A','B','X'], ['A','B','C','D']));`}
