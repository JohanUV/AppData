## What is Git and why does it matter?

Picture this: you've spent three days polishing a Python script that cleans a
sales dataset. It works. You decide to "improve" it with a quick refactor and
20 minutes later nothing compiles, you can't remember what you changed, and
the working version vanished between Ctrl+Z and coffee. That scene has a name:
**working without version control**. And it's exactly the problem Git solves.

**Git** is a distributed version control system. Its job is concrete: record
every change in your files, let you go back to any previous state, and let
several people work in parallel without stepping on each other. It isn't just
another tool in the Data Engineer's belt — it's the assumed tool, the one
that separates code that can be shared from code that can't.

<Callout type="info" title="Key idea">
  Git doesn't store diffs. It stores **complete snapshots** of the project at
  every commit, deduplicated by hash. That's why switching branches is
  instantaneous: it just points to another snapshot.
</Callout>

### The problem Git solves

Before Git there were alternatives: copy folders with suffixes
(`pipeline_v3_FINAL_FINAL`), email zips around, or use centralized tools like
SVN. Each had a bottleneck: expensive to coordinate, dependent on a permanent
central server, or unable to work offline.

Linus Torvalds wrote Git in 2005 with three non-negotiable requirements:

1. **Distributed** — every local copy is a complete repository, with full
   history. You don't depend on a server to commit.
2. **Fast** — local operations in milliseconds, even with huge repos.
3. **Integrity** — every commit is identified with a SHA-1 hash; changing one
   byte of history changes every subsequent hash. Tampering is impossible
   silently.

### How Git fits into Data Engineering

For a Data Engineer, Git is the backbone of daily work:

- **Pipelines as code**: your Airflow DAGs, dbt models, Spark scripts — all
  live in a repo. Without Git, "deploy" means "I FTP'd the file and prayed."
- **Reproducibility**: commit `abc123` defines exactly which code ran the
  pipeline at 3am Tuesday. If data went sideways, you go to that exact state
  and debug.
- **Code review**: the moment you learn the most is when someone reads your PR
  and says "this breaks if the API returns null." Without Git, no PR.
- **Rollback**: a release breaks production? `git revert` puts you back in
  30 seconds.

### The mental model: three areas

<Diagram type="mermaid">
{`flowchart LR
  A[Working Directory<br/>editable files] -->|git add| B[Staging Area<br/>changes ready]
  B -->|git commit| C[Repository<br/>immutable history]
  C -->|git checkout| A`}
</Diagram>

Git divides your project into three "zones":

- **Working directory**: what you see in your folder. You edit files here.
- **Staging area** (index): the "draft" of the next commit. Files marked with
  `git add` wait here.
- **Repository**: the permanent history. When you `git commit`, what's in
  staging is sealed into a new snapshot.

This separation is what enables **atomic** commits: only related work goes in
together. If you touched 5 files but only 3 are part of the bug you fixed,
you `git add` only those 3 — the rest stay for another commit.

### Distributed vs centralized

In Subversion (SVN) there was a central server. Every commit went straight
there; no network, no commits. Git inverts the model: your local copy **is** a
full repository. You can do 50 commits on a plane with no connection, then
sync with `git push` and `git pull` once you land.

This changes how you work: experiment locally without fear, do three trial
commits, squash them into a clean one and only then share. The "narrative" of
history is built locally, not on the server.

### Common mistakes

- **Confusing Git with GitHub**. Git is the tool; GitHub is a service that
  hosts Git repositories in the cloud (like GitLab or Bitbucket). You can use
  Git without GitHub just fine.
- **One giant commit with "updated everything"**. You lose the ability to
  revert a specific change. Small, themed commits, always.
- **Versioning generated files** (logs, output `.parquet`, `__pycache__`).
  They bloat the repo and add no value; that's what `.gitignore` is for
  (lesson 8).
- **Thinking `git push` "saves"**. Every commit is already saved locally.
  `git push` only syncs with a remote.

## Practical exercise

<Exercise
  id="git-what-and-why-exercise"
  title="Identify the zone"
  description="Given a sequence of actions, return which zone the change lives in after each step. The function takes a list of commands and returns the resulting zone."
  language="javascript"
  starterCode={`function whereIsTheChange(commands) {
  // commands is an array like ['edit', 'add', 'commit']
  // 'edit' -> 'working'
  // 'add'  -> 'staging'
  // 'commit' -> 'repo'
  // Return the zone after the last command.
  return 'working';
}

const answer = whereIsTheChange(['edit', 'add', 'commit']);
console.log('final zone:', answer);`}
  solution={`function whereIsTheChange(commands) {
  const map = { edit: 'working', add: 'staging', commit: 'repo' };
  const last = commands[commands.length - 1];
  return map[last] ?? 'working';
}
const answer = whereIsTheChange(['edit', 'add', 'commit']);
console.log('final zone:', answer);`}
  hints={[
    'Only the last command in the list matters.',
    'Use an object as a lookup table: { edit, add, commit }.',
  ]}
  tests={[
    { description: 'edit → working', assert: "whereIsTheChange(['edit']) === 'working'" },
    { description: 'edit + add → staging', assert: "whereIsTheChange(['edit','add']) === 'staging'" },
    { description: 'full flow → repo', assert: "whereIsTheChange(['edit','add','commit']) === 'repo'" },
  ]}
/>
