## Trabalhando com GitHub remoto

Até aqui o Git viveu só na sua máquina. Já é valioso, mas o poder real
chega quando você conecta a um **remoto**: um repositório Git hospedado em
outro lugar (GitHub, GitLab, Bitbucket). O remoto é o ponto de
sincronização entre devs e o backup do seu código se o disco morrer.

Nesta lição você vincula o repo local ao GitHub, aprende a sincronizar com
`push`, `pull` e `fetch`, e configura SSH para não pedir senha em cada
operação.

### Criar repo no GitHub

1. [github.com](https://github.com), faça login.
2. Canto superior direito → **+** → **New repository**.
3. Nome (`meu-primeiro-pipeline`), descrição opcional, público/privado.
4. **Não marque** "Add a README file" — você já tem um local.
5. Crie.

GitHub mostra instruções. As que importam (caso "push an existing
repository"):

```bash
git remote add origin git@github.com:seu-user/meu-primeiro-pipeline.git
git branch -M main
git push -u origin main
```

Antes de rodar, configure SSH.

### HTTPS vs SSH

- **HTTPS** — `https://github.com/user/repo.git`. Pede token a cada push
  (senhas não aceitas desde 2021). Fácil com credential helper.
- **SSH** — `git@github.com:user/repo.git`. Uma vez configurado, sem
  prompts. Mais cômodo no longo prazo.

Vamos com SSH (padrão em tutoriais sérios).

### Configurar SSH passo a passo

```bash
# 1. Gerar chave
ssh-keygen -t ed25519 -C "voce@email.com"

# 2. Iniciar agente e adicionar chave
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# 3. Copiar chave pública
cat ~/.ssh/id_ed25519.pub
# macOS: pbcopy < ~/.ssh/id_ed25519.pub
# Windows Git Bash: cat ~/.ssh/id_ed25519.pub | clip
```

**4. Cole a chave no GitHub:** Settings → SSH and GPG keys → **New SSH
key** → Cole → Save.

**5. Verificar:**

```bash
ssh -T git@github.com
# Hi seu-user! You've successfully authenticated...
```

### Conectar local com remoto

```bash
git remote add origin git@github.com:seu-user/meu-primeiro-pipeline.git
git remote -v
git branch -M main
git push -u origin main
```

`-u` faz a branch local lembrar do par remoto. Depois `git push` e `git
pull` sem args funcionam.

### Fluxo típico

```
manhã:    git pull origin main          # Começo atualizado
meio:     git push origin feature/x     # Sobe trabalho
tarde:    git fetch && git log ...      # Vejo o que outros fizeram
```

### Clonar repo existente

```bash
git clone git@github.com:org/projeto.git
cd projeto
git remote -v       # já configurado
git branch -a       # branches locais e remotas
```

### Push rejeitado: como arrumar

```
! [rejected]        main -> main (fetch first)
```

Significa: alguém pushed algo novo a main que você não tem.

```bash
git pull --rebase
git push
```

`--rebase` mantém histórico linear (sem merge commits "Merge branch
main"). Boa prática para branches solo.

### Erros comuns

- **`git push --force` para "consertar"**. Sobrescreve o remoto e destrói
  trabalho de outros. Só em branches pessoais e nunca em main.
- **Esquecer `-u` no primeiro push**.
- **Confundir fetch com pull**.
- **Pushear segredos**. Crawlers indexam repos públicos em minutos.
  Considere a chave comprometida e rote.
- **Usar HTTPS pedindo senha sempre**. Configure credential helper ou
  passe pra SSH.

  solution={`function parseGithubUrl(url) {
  const ssh = url.match(/^git@github\\.com:([^/]+)\\/(.+?)(?:\\.git)?$/);

  const https = url.match(/^https:\\/\\/github\\.com\\/([^/]+)\\/(.+?)(?:\\.git)?$/);
  if (https) return { owner: https[1], repo: https[2] };
  return null;
