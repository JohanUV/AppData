## Best practices and .gitignore

You close this module with the habits that separate you from the casual user.
Knowing commands is 30% of the job; the other 70% is discipline using them:
what message you write, what files you ignore, when you branch. This lesson
collects the highest-leverage practices.

### Commit messages worth writing

A commit is a small change unit + a message. The **message is for humans**:
the `git log` your team will read 6 months later when something breaks.

**Minimum rules:**

1. **First line ≤ 72 chars.** The subject — what shows up in `git log
   --oneline`. Imperative, present tense: "add" not "adding" or "added".
2. **Blank line** between subject and body (if there's a body).
3. **Optional body** explaining the why when not obvious. ~72 chars per line.
4. **Semantic prefix** (Conventional Commits): `feat`, `fix`, `docs`,
   `refactor`, `test`, `chore`, `perf`, `build`, `ci`.

Full example:

```
fix(pipeline): handle null amount in payments aggregation

Stripe webhook can send amount=null when a charge is voided
mid-processing. Previously this crashed the daily aggregation
with TypeError. Now we coalesce null to 0 and log a warning.

Refs: DATA-142
```

Why does it matter? Because 6 months from now, debugging:

```bash
git log --oneline --grep="payment"
# fix(pipeline): handle null amount in payments aggregation
# feat(pipeline): add payments aggregation
```

You see immediately what to look at. Without convention, that log is
unreadable.

### Atomic commits

A commit should represent **one coherent idea**. If your message contains
"and", chances are it's two commits.

Bad:

```
fix bug + add feature + update README
```

Good (3 commits):

```
fix(api): retry on 5xx with exponential backoff
feat(metrics): add request latency histogram
docs: update setup instructions for new metrics
```

Benefits:
- `git revert` undoes one change without touching the others.
- `git bisect` (bisection debugging) pinpoints the bug-introducing commit.
- Review is focused.

### .gitignore: what should never enter the repo

`.gitignore` is a file (or several, nestable in subfolders) listing patterns
Git must **fully ignore**. Matching files don't appear in `git status` and
can't be added by accident.

Categories to ignore:

**1. Generated / builds:**
```
dist/
build/
out/
*.pyc
__pycache__/
node_modules/
.next/
target/
```

**2. IDE / system:**
```
.vscode/
.idea/
.DS_Store
Thumbs.db
```

**3. Logs and temp data:**
```
*.log
logs/
*.tmp
.cache/
```

**4. Raw data / outputs (Data Engineering):**
```
data/raw/
data/processed/
*.parquet
*.csv
*.xlsx
```

**5. Secrets and local config:**
```
.env
.env.local
*.pem
*.key
secrets.json
config.local.yaml
```

Secrets are **the most critical**. An AWS key committed to a public repo is
discovered in under 5 minutes by bots watching GitHub. If it happens,
consider the key compromised and rotate it immediately — deleting the
commit is not enough.

### .gitignore templates

GitHub maintains official per-language templates:

```bash
curl https://raw.githubusercontent.com/github/gitignore/main/Python.gitignore > .gitignore
```

There are templates for Python, Node, Java, Go, etc. Start with one and add
project-specifics.

### If you already committed something you shouldn't have

Common case: you pushed a `.env` with a key. Steps:

1. **Rotate the credential immediately** in the provider (AWS, Stripe, ...).
   Assume it's compromised.
2. Add `.env` to `.gitignore`.
3. Remove the file from the repo (without deleting locally):
   ```bash
   git rm --cached .env
   git commit -m "chore: remove .env from tracking"
   git push
   ```
4. To remove from **full history** (not just last commit), use
   `git filter-repo` or `BFG Repo Cleaner`. More invasive, rewrites
   history — coordinate with the team.

Even after deleting from history, the credential is compromised. Step 1 is
non-negotiable.

### Other best practices

- **Name branches with prefix and kebab-case**: `feature/null-handling`,
  `fix/api-timeout`, `chore/upgrade-deps`. Some teams use
  `your-user/feature-x` to identify authorship.
- **Rebase long-running branches frequently** onto `main` to avoid pile-up:
  ```bash
  git switch feature/long-running
  git fetch origin
  git rebase origin/main
  ```
- **A whole career using `git pull --rebase` by default**:
  ```bash
  git config --global pull.rebase true
  ```
  Keeps history linear instead of generating merge commits per pull.
- **No WIP commits on `main`**. Use a branch or, if you need to pause fast,
  `git stash`.
- **Enable signed commits** if your org requires them (GPG or SSH). Proves
  the commit was authorized by you.

### Common mistakes

- **Repos with `data/`, `node_modules/`, `__pycache__/` versioned**. The
  repo is 2GB, pulls are slow, diffs are unreadable.
- **Commit messages "wip" or "fix"**. Convey nothing. Same as no message.
- **`force-push` to `main`**. Wipes others' work without warning. Even on
  "your" repo, don't do it — you'll forget once and pay later.
- **6-month-old branch never rebased**. Merging is conflict hell. Short
  branches always.
- **Deleting `.git/`**. You lose all history. No going back. Yes, it
  happens more than you'd think.

## Practical exercise

<Exercise
  id="git-best-practices-exercise"
  title="Filter files by .gitignore"
  description="Implement `shouldIgnore(path, patterns)` returning true if path matches any of the simple patterns. Supports trailing / for directories and * as a wildcard for non-/ characters."
  language="javascript"
  starterCode={`function shouldIgnore(path, patterns) {
  return false;
}

console.log(shouldIgnore('src/__pycache__/foo.pyc', ['__pycache__/']));
console.log(shouldIgnore('app.log', ['*.log']));
console.log(shouldIgnore('app.py', ['*.log']));`}
  solution={`function shouldIgnore(path, patterns) {
  for (const p of patterns) {
    if (p.endsWith('/')) {
      const dirName = p.slice(0, -1);
      if (path.split('/').includes(dirName)) return true;
    } else if (p.startsWith('*.')) {
      const ext = p.slice(1);
      if (path.endsWith(ext)) return true;
    } else {
      if (path === p || path.endsWith('/' + p)) return true;
    }
  }
  return false;
}
console.log(shouldIgnore('src/__pycache__/foo.pyc', ['__pycache__/']));
console.log(shouldIgnore('app.log', ['*.log']));
console.log(shouldIgnore('app.py', ['*.log']));`}
  hints={[
    'For patterns ending in /, check if any segment of the path equals that directory.',
    'For *.ext, path.endsWith(".ext") suffices.',
    'For exact names like .env, compare equality or trailing /.env.',
  ]}
  tests={[
    {
      description: 'nested directory',
      assert: 'shouldIgnore("a/b/__pycache__/c.pyc", ["__pycache__/"]) === true',
    },
    {
      description: 'extension wildcard',
      assert: 'shouldIgnore("logs/app.log", ["*.log"]) === true',
    },
    {
      description: 'exact file',
      assert: 'shouldIgnore(".env", [".env"]) === true',
    },
    {
      description: 'no match',
      assert: 'shouldIgnore("src/main.py", ["*.log", "node_modules/"]) === false',
    },
  ]}
/>
