## Installation and initial setup

Before making any commits you need Git installed and you need to tell it who
you are. Sounds obvious, but this is where 80% of beginners trip up: they
install Git, make a commit, and the author shows up as `unknown@DESKTOP-XYZ`
in history. That's hard to scrub later. Better to set it up right from the
start.

In this lesson we will:

1. Install Git on your OS.
2. Configure your identity (name, email).
3. Pick a default editor.
4. Verify everything with `git --version`.

### Installing per OS

**Windows.** Download the official installer from
[git-scm.com/download/win](https://git-scm.com/download/win). It's an `.exe`
that bundles Git Bash, a small Linux-style terminal you'll need (PowerShell
also works, but Git Bash is closer to the Unix ecosystem Git was born in).
Most defaults during install are fine, with two notes:

- **Default editor**: pick one you actually know. VS Code is a good choice if
  you already use it.
- **Line endings**: pick "Checkout Windows-style, commit Unix-style". Git
  will normalize line endings so repos don't get phantom diffs when
  collaborating across macOS/Linux.

**macOS.** Three paths:

```bash
# Option A: comes with Xcode Command Line Tools
xcode-select --install

# Option B: Homebrew (more up-to-date)
brew install git

# Option C: official installer from git-scm.com
```

**Linux.** Use your package manager:

```bash
# Debian / Ubuntu
sudo apt update && sudo apt install git

# Fedora / RHEL
sudo dnf install git

# Arch
sudo pacman -S git
```

### Verify the install

Open your terminal (Git Bash on Windows, Terminal on macOS, any emulator on
Linux) and run:

```bash
git --version
```

You should see something like `git version 2.45.2`. Anything ≥ 2.30 is fine.
If you get "command not found", restart the terminal (PATH variables don't
always reload). If it persists, check that the install added Git to PATH.

### Configure your identity

Every commit carries a name and an email. These aren't authenticated (Git
trusts you locally) but they're recorded in history forever. Configure the
globals once:

### Useful extras

These aren't required but will save you pain:

```bash
# Better colored log output
git config --global color.ui auto

# Aliases many people use
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.lg "log --oneline --graph --all"

# Resolve conflicts in your editor
git config --global merge.tool vscode
```

With this, `git st` equals `git status` and `git lg` shows the visual repo
history one commit per line.

### Where the config lives

Git has **three config levels**, in precedence order:

1. **System** (`/etc/gitconfig`) — for every user. Rarely touched.
2. **Global** (`~/.gitconfig`) — for your user. Identity goes here.
3. **Local** (`.git/config` inside the repo) — project-specific.

If your work and personal accounts use different emails, override per repo:

```bash
cd ~/projects/work-thing
git config user.email "you@company.com"
```

That only affects that repo, leaving the global intact for everything else.

### Common mistakes

- **Forgetting `--global`** and then wondering why every repo asks for your
  name again. If you're configuring personal identity, you almost always
  want `--global`.
- **Email different from GitHub's**: your commits don't link to your profile.
  Always use a verified email.
- **Notepad as default Windows editor**: it kills multi-line commit messages
  with weird encodings. Prefer `code --wait`, `nano` or `vim`.
- **Skipping line endings on Windows** and fighting invisible diffs later.
  `core.autocrlf=true` is the right behavior.

  solution={`function buildConfigCmd(key, value) {

const b = buildConfigCmd('color.ui', 'auto');
console.log(a);
console.log(b);`}
