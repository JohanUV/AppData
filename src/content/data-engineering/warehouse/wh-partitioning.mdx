## Particionamiento y clustering

En warehouses columnares, **particionamiento** y **clustering** son
las dos palancas más importantes para performance y costo. Sin ellos,
cada query escanea toda la tabla. Con ellos, solo lo necesario.

### Particionamiento: dividir en chunks físicos

Particionar = dividir una tabla en archivos separados según una
columna. El motor solo lee particiones relevantes (partition pruning).

**BigQuery:**

```sql
CREATE TABLE analytics.events (
  event_id STRING,
  user_id STRING,
  event_type STRING,
  amount NUMERIC,
  created_at TIMESTAMP
)
PARTITION BY DATE(created_at);
```

Esa tabla se almacena físicamente como una carpeta por día:

```
events/
├── 2026-01-01/data.parquet
├── 2026-01-02/data.parquet
└── ...
```

Una query con `WHERE DATE(created_at) = '2026-01-15'` toca **un**
archivo, no 365.

**Snowflake** usa **micro-partitions** automáticas (16 MB cada una)
ordenadas por insertion order. No defines particionamiento manual; lo
manejas con clustering.

**Redshift** usa `DISTSTYLE` y `SORTKEY` (similar pero diferente
modelo).

### Cuál columna usar para particionar

Reglas:

1. **Columna usada en `WHERE` casi siempre**. Para event data: fecha.
   Para multi-tenant: tenant_id.
2. **Cardinalidad moderada**. 365 días/año = OK. 1 millón de
   timestamps únicos = explosión.
3. **Truncar a granularidad correcta**: `DATE(timestamp)`, no
   `timestamp`.

Mala particionamiento es peor que sin particionamiento — overhead de
metadata sin beneficio.

### Clustering: orden físico dentro de partición

Clustering ordena los datos dentro de cada archivo por columnas
clave. Como un índice secundario adaptado a columnar.

**BigQuery:**

```sql
CREATE TABLE analytics.events (...)
PARTITION BY DATE(created_at)
CLUSTER BY user_id, event_type;
```

Dentro de cada partición de fecha, los datos se almacenan ordenados
por (user_id, event_type). Una query con `WHERE user_id = 'U123'`
salta directamente a las páginas relevantes.

**Snowflake:**

```sql
CREATE TABLE events (...) CLUSTER BY (user_id, event_type);
-- O alterar:
ALTER TABLE events CLUSTER BY (user_id, event_type);
```

Snowflake re-clusteriza automáticamente con un servicio de fondo
(Auto-Clustering — pago aparte).

### Order matters

```sql
CLUSTER BY user_id, event_type
```

Funciona bien para:

- `WHERE user_id = X`
- `WHERE user_id = X AND event_type = Y`

NO bien para:

- `WHERE event_type = Y` (solo, sin user_id) — eficiente parcialmente

Pon primero la columna más selectiva o la usada con más frecuencia
sola.

### Verificar que funcione

**BigQuery:** la UI te muestra "bytes processed" antes de ejecutar.
Sin partition pruning, ves TBs; con pruning, MBs.

**Snowflake:** mira el query profile en Snowsight: "partitions
scanned" / "partitions total". Idealmente menor al 10%.

### Anti-patrones

```sql
-- ❌ Particionar por timestamp exacto (alta cardinalidad)
PARTITION BY created_at        -- millones de particiones

-- ❌ Filtrar por columna distinta a la de partición
WHERE EXTRACT(YEAR FROM created_at) = 2026   -- no usa pruning
-- ✓ usa función explícita compatible
WHERE created_at >= '2026-01-01' AND created_at < '2027-01-01'

-- ❌ Cluster por columnas no usadas en filtros
CLUSTER BY event_id           -- siempre único, sirve para nada

-- ❌ Demasiadas columnas en CLUSTER BY
CLUSTER BY a, b, c, d, e, f    -- pierde efectividad después de 2-4
```

### Cuándo no particionar

- **Tablas pequeñas** (menores a 1GB): overhead de metadata pesa más que el
  ahorro.
- **Sin columna obvia para particionar**: si las queries no comparten
  un filtro común, particionar arbitrariamente puede empeorar.
- **Snowflake tablas menores a 1TB**: las micro-partitions automáticas suelen
  bastar; clusterizar tiene costo.

### Errores comunes

- **Olvidar particionar tablas TB-scale**: factura mensual ridícula.
- **Particionar por timestamp exacto**: explosión.
- **`CLUSTER BY event_id`**: sin selectividad.
- **Filtrar con función que rompe pruning**: usa columna directa.
- **No verificar bytes scanned**: optimizas a ciegas.
