## BigQuery hands-on

**BigQuery** es el warehouse serverless de Google Cloud. No manejas
clusters, nodos ni tuning: levantas, escribes SQL, paga por bytes
escaneados. Para Data Engineering en GCP es la opción default;
también lo verás en stacks multi-cloud por su ergonomía.

### Free tier para practicar

- 1 TB de queries escaneados/mes — gratis.
- 10 GB de storage — gratis.
- Suficiente para seguir esta lección y experimentar mucho.

[console.cloud.google.com/bigquery](https://console.cloud.google.com/bigquery)

### Estructura de objetos

```
proyecto-gcp
└── dataset
    ├── table
    └── view
```

- **Project**: tu cuenta GCP. Cobranza y permisos.
- **Dataset**: agrupamiento lógico (~ schema en otros motores).
- **Table / View / Materialized View**: como en SQL clásico.

### SQL: standard, no dialecto raro

```sql
SELECT
  date_trunc(date(created_at), MONTH) AS month,
  country,
  count(distinct user_id) AS active_users,
  sum(amount) AS revenue
FROM `my-project.analytics.events`
WHERE created_at >= '2026-01-01'
  AND event_type = 'purchase'
GROUP BY 1, 2
ORDER BY 1, revenue DESC;
```

Backticks para nombres con `-` o `.`. SQL moderno: window functions,
CTEs, arrays nativos, JSON nativo.

### Cargar datos

**Desde GCS (lo más eficiente):**

```sql
LOAD DATA INTO analytics.orders
FROM FILES (
  format = 'PARQUET',
  uris = ['gs://my-bucket/orders/*.parquet']
);
```

**Desde la UI o `bq` CLI:**

```bash
bq load --source_format=PARQUET \
  analytics.orders \
  gs://my-bucket/orders/*.parquet
```

**Desde Python:**

```python
from google.cloud import bigquery

client = bigquery.Client()
job = client.load_table_from_uri(
    'gs://my-bucket/orders/*.parquet',
    'analytics.orders',
    job_config=bigquery.LoadJobConfig(
        source_format=bigquery.SourceFormat.PARQUET,
        autodetect=True,
    ),
)
job.result()
```

### Particionamiento

Particionar es lo más importante para rendimiento y costo en BigQuery.

```sql
CREATE TABLE analytics.events (
  event_id STRING,
  user_id STRING,
  event_type STRING,
  amount NUMERIC,
  created_at TIMESTAMP
)
PARTITION BY DATE(created_at)
CLUSTER BY user_id, event_type;
```

Cada query con `WHERE created_at >= 'X'` solo escanea las particiones
relevantes:

### Slots: la moneda del compute

BigQuery factura compute en **slots** (unidades de paralelismo). Dos
modelos:

- **On-demand** (default): pagas por bytes escaneados. Sin compromiso.
- **Reservations / Editions**: compras slots fijos. Predicible, mejor
  para workloads constantes.

Para empezar y experimentar: on-demand. Cuando consumes >1TB/día
constante: evalúa reservations.

### Datasets públicos

BigQuery hosta datasets públicos enormes (Wikipedia, GitHub, NYC taxi):

```sql
-- Cuenta archivos por lenguaje en GitHub
SELECT lang, COUNT(*) AS files
FROM `bigquery-public-data.github_repos.languages`,
UNNEST(language) AS l, l.lang AS lang
GROUP BY 1
ORDER BY 2 DESC
LIMIT 10;
```

Perfecto para practicar SQL sobre TBs.

### Funciones útiles únicas de BigQuery

- **ARRAY / STRUCT**: tipos nidificados nativos.
  ```sql
  SELECT user_id, items[OFFSET(0)].sku FROM orders;
  ```
- **UNNEST**: desplegar arrays a filas.
  ```sql
  SELECT user_id, item.sku
  FROM orders, UNNEST(items) AS item;
  ```
- **APPROX_COUNT_DISTINCT**: HyperLogLog, mucho más rápido.
- **ML SQL** (BQML): entrena modelos sin salir de BQ.
  ```sql
  CREATE MODEL analytics.churn
  OPTIONS(model_type='logistic_reg') AS
  SELECT * FROM analytics.user_features;
  ```

### Best practices

- **Particiona por DATE(timestamp)**: casi todas las tablas grandes.
- **Cluster por columnas frecuentes en WHERE**: 1-4 columnas (no más).
- **Evita `SELECT *`**: cobras cada columna que tocas.
- **Pre-agregar a materialized views**: cosas que se consultan
  diariamente.
- **Pre-vista cost en UI antes de ejecutar**: la barra superior te dice
  bytes a procesar.

### Errores comunes

- **No particionar tablas TB-scale**: cada query escanea todo →
  factura mensual ridícula.
- **Particionar por columnas con alta cardinalidad** (timestamp
  exacto, user_id): explosión de particiones.
- **`SELECT *` por costumbre**: trae 50 columnas, paga 50.
- **Streaming inserts (sin `STREAM`)** sin throttling: caro y lento
  comparado a batch.
- **No usar legacy SQL en error**: si ves `#legacySQL` te encuentras
  en sintaxis antigua, cámbiate a Standard SQL.
