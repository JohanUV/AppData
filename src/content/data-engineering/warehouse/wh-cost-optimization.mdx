## Cost optimization

Los warehouses cloud son cómodos hasta que llega la primera factura
sorprendente. Equipos pequeños llegan a gastar $50k/mes sin darse
cuenta. Esta lección cubre los lever más impactantes para mantener el
gasto bajo.

### Anatomía de la factura

Casi todos cobran por:

1. **Storage** ($/GB/mes). Barato. ~$20/TB/mes.
2. **Compute**:
   - BigQuery: $5/TB **escaneado** (on-demand) o slots reservados.
   - Snowflake: $X/hora cuando warehouse activo.
   - Redshift: $/hora del cluster.
3. **Egress**: sacar datos al exterior.
4. **Streaming inserts** (BQ, otros): premium sobre batch loads.

Storage rara vez es el problema. Compute lo es.

### Top palancas (ordenadas por impacto)

#### 1. Particionar y clusterizar

Cubierto en lección anterior. Mejor inversión de tiempo: 1 hora de
trabajo, 80% de ahorro en queries grandes.

#### 2. Materializar resultados frecuentes

Una query analítica corre 100 veces al día. Si tarda 30s y escanea
500GB, son ~5 horas de compute y 50TB escaneados/día.

Solución: pre-computar como tabla incremental con dbt.

```sql
-- Antes: cada dashboard re-calcula
SELECT date, country, SUM(amount) FROM events
WHERE date >= today() - 30
GROUP BY 1, 2;

-- Después: dbt incremental table; dashboards leen pre-agregado
{{ config(materialized='incremental', unique_key=['date','country']) }}
SELECT date, country, SUM(amount) AS revenue FROM events ...
```

10× menos costo, queries instantáneas.

#### 3. Eliminar `SELECT *`

En columnar, cada columna leída cuesta. Tabla con 100 columnas; query
necesita 5; `SELECT *` paga 20× más. Sin razón.

#### 4. Auto-suspend agresivo (Snowflake)

```sql
ALTER WAREHOUSE etl_wh SET AUTO_SUSPEND = 60;
```

Si tu warehouse corre 5 min/hora pero tiene auto-suspend de 10 min,
pagas casi todo el día.

#### 5. Escalado correcto

Snowflake: warehouses chicos para queries chicas. Subir a XL solo
para batches grandes ad-hoc, no para BI dashboards.

BigQuery: en on-demand no aplica; en reservations, ajusta slots por
proyecto.

#### 6. Incremental en lugar de full-refresh

dbt incremental:

```sql
{{ config(materialized='incremental') }}
SELECT * FROM source
{% if is_incremental() %}
WHERE updated_at > (SELECT MAX(updated_at) FROM {{ this }})
{% endif %}
```

Procesar 1M filas nuevas vs 100M históricas = 100× menos costo.

#### 7. Time-travel y storage

Snowflake mantiene tablas históricas (Time Travel) hasta 90 días.
Cobra storage doble durante ese período. Si no necesitas tanto:

```sql
ALTER TABLE big_table SET DATA_RETENTION_TIME_IN_DAYS = 1;
```

#### 8. Drop / archive cold data

Tabla con 5 años de eventos. ¿Realmente consultas data de hace 4
años? Considera moverla a S3 cold storage o eliminarla.

```sql
DELETE FROM events WHERE date < CURRENT_DATE - 730;
-- O mejor: particionar y droppear partition vieja
```

### Quotas y alerts

Configura **límites de gasto** desde el día 1:

- BigQuery: Cuotas por proyecto/usuario en bytes/día.
- Snowflake: Resource Monitors con notificaciones a 50%, 75%, 100% del
  budget.

```sql
CREATE RESOURCE MONITOR monthly_budget
  WITH CREDIT_QUOTA = 1000
  FREQUENCY = MONTHLY
  TRIGGERS
    ON 75 PERCENT DO NOTIFY
    ON 90 PERCENT DO SUSPEND
    ON 100 PERCENT DO SUSPEND_IMMEDIATE;
```

### Errores comunes

- **Sin alerts ni quotas**: factura sorpresa al final del mes.
- **Optimizar sin medir**: cambias cosas a ciegas. Mide antes y
  después.
- **`SELECT *` en producción** sin nadie revisar.
- **Materialized views sin refresh adecuado**: pagas el storage y el
  refresh sin beneficio.
- **Tablas históricas inmensas que nadie consulta**: paga storage
  para nada.

## Ejercicio práctico

<Exercise
  id="wh-cost-exercise"
  title="Estimador de ahorro por particionamiento"
  description="Implementa `savingsByPartitioning(totalBytes, daysQueried, totalDays)` que devuelve % de ahorro. Si filtras 30 de 365 días, ahorras 91.7%."
  language="javascript"
  starterCode={`function savingsByPartitioning(totalBytes, daysQueried, totalDays) {
  return 0;
}

console.log(savingsByPartitioning(1e12, 30, 365));   // ~0.918`}
  solution={`function savingsByPartitioning(totalBytes, daysQueried, totalDays) {
  if (totalDays <= 0) return 0;
  return 1 - (daysQueried / totalDays);
}
console.log(savingsByPartitioning(1e12, 30, 365));`}
  hints={[
    'savings = 1 - (queried / total).',
  ]}
  tests={[
    { description: 'mes de año', assert: 'Math.abs(savingsByPartitioning(1e12, 30, 365) - (1 - 30/365)) < 1e-9' },
    { description: 'todo escaneado = 0 ahorro', assert: 'savingsByPartitioning(1e12, 365, 365) === 0' },
    { description: 'nada escaneado = 100%', assert: 'savingsByPartitioning(1e12, 0, 365) === 1' },
  ]}
