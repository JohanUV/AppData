## Reverse ETL

**Reverse ETL** es el patrón de mover datos **del warehouse hacia
sistemas operacionales** — el inverso del ETL clásico. Tu CRM,
herramienta de marketing, o producto necesita los datos transformados
que viven en el warehouse. Reverse ETL es el puente.

### Por qué existe

ETL tradicional:

```
Sistemas operacionales → Warehouse → BI / Dashboards
```

Reverse ETL:

```
Warehouse → Sistemas operacionales (Salesforce, HubSpot, Intercom, ...)
```

¿Para qué? **Activación de datos**: los datos que están en el warehouse
limpios, segmentados, enriquecidos también deberían estar en las
herramientas que el equipo de marketing/ventas/producto usa día a día.

### Casos de uso

- **Customer 360 → CRM**: warehouse calculó el LTV, churn risk, last
  order date. Push esos campos a Salesforce para que el equipo de
  ventas los vea.
- **Audiencias para marketing**: "usuarios premium con >5 órdenes en
  los últimos 30 días" → push a HubSpot/Mailchimp para campañas.
- **Personalization**: features de ML (predicted_category) →
  Intercom/in-app messaging.
- **Customer support**: conversation_count, last_purchase →
  Zendesk/Intercom para que support vea contexto.

El **modelo de datos** vive en el warehouse (single source of truth).
La sincronización a sistemas externos es una capa adicional, no el
master data.

### Las herramientas

- **Hightouch**: una de las más usadas. UI para definir audiencias y
  destinos sin código.
- **Census**: similar, también muy adoptado.
- **Polytomic**: orientado a developers.
- **Reverse ETL custom**: scripts Python con APIs de cada destino. Ok
  para un destino, pesadilla a escala.

Las managed son la opción default por el costo de mantener
integraciones a docenas de SaaS.

### Cómo funciona conceptualmente

1. **Defines el modelo en warehouse** (típicamente con dbt):
   ```sql
   -- models/marts/customer_enriched.sql
   SELECT
     customer_id,
     email,
     ltv,
     churn_risk,
     last_order_date,
     segment
   FROM {{ ref('dim_customer') }}
   JOIN {{ ref('fct_customer_metrics') }} USING (customer_id);
   ```

2. **Mapeas a campos del destino** (Salesforce, HubSpot...):
   - `customer_enriched.email` → `Salesforce.Contact.Email`
   - `customer_enriched.ltv` → `Salesforce.Contact.LTV__c`
   - `customer_enriched.segment` → `Salesforce.Contact.Segment__c`

3. **Defines schedule + sync mode**:
   - **Full sync**: enviar todo cada vez (caro, lento).
   - **Incremental sync**: solo registros cambiados.
   - **Mirror sync**: refleja exactamente el estado del warehouse
     (incluyendo deletes).

4. **El servicio detecta cambios y los empuja vía API** del destino.

### Consideraciones

- **Rate limits del destino**: APIs SaaS típicamente 10-100 req/s.
  Sincronizar 1M registros toma horas. La herramienta gestiona esto.
- **Idempotencia**: si una sincronización falla a la mitad, retry sin
  duplicar. Las managed lo manejan; en custom, debes diseñarlo.
- **Schema evolution**: agregas un campo nuevo al modelo → necesitas
  agregarlo en el destino (custom field en Salesforce). Coordina con
  los equipos owners del destino.
- **Privacidad**: el destino puede ser una herramienta SaaS de
  terceros. Asegura que solo envías datos permitidos (no PII a
  herramientas no autorizadas).

### Cuándo NO usar Reverse ETL

- **Latencia sub-segundo**: reverse ETL es batch (5-60 min típico). Para
  realtime usa CDC + streaming.
- **Datos que NO están limpios en warehouse**: no tiene sentido
  sincronizar basura.
- **El destino ya recibe los datos directamente**: por ejemplo, Stripe
  ya tiene tu data de pagos — no se la mandes desde el warehouse.

### Errores comunes

- **Sincronizar mucho, mucho más de lo necesario**: rate limits, costo
  alto. Filtra por audiencia.
- **Sin tracking column**: full sync cada vez = ineficiente.
- **Owner ambiguo**: cuando el campo en Salesforce está mal, ¿quién lo
  arregla — analytics o sales ops? Define quién es source of truth.
- **Sync schedule muy frecuente**: cada 5 min cuando los datos
  cambian cada hora = waste.

## Ejercicio práctico

<Exercise
  id="wh-reverse-etl-exercise"
  title="Mapper de campos"
  description="Implementa `mapFields(rows, mapping)` donde rows es array de objetos y mapping un objeto { sourceField: destField }. Devuelve array transformado."
  language="javascript"
  starterCode={`function mapFields(rows, mapping) {
  return [];
}

const rows = [{ customer_id: 1, ltv: 100 }];
const mapping = { customer_id: 'Id', ltv: 'LTV__c' };
console.log(mapFields(rows, mapping));
// [{ Id: 1, LTV__c: 100 }]`}
  solution={`function mapFields(rows, mapping) {
  return rows.map(r => {
    const out = {};
    for (const [src, dest] of Object.entries(mapping)) {
      if (src in r) out[dest] = r[src];
    }
    return out;
  });
}
const rows = [{ customer_id: 1, ltv: 100 }];
const mapping = { customer_id: 'Id', ltv: 'LTV__c' };
console.log(mapFields(rows, mapping));`}
  hints={[
    'map cada row, construye un nuevo objeto con las claves renombradas.',
  ]}
  tests={[
    {
      description: 'rename simple',
      assert: 'JSON.stringify(mapFields([{a:1}],{a:"X"})) === \\\'[{"X":1}]\\\'',
    },
    {
      description: 'campos sin mapping se ignoran',
      assert: 'Object.keys(mapFields([{a:1,b:2}],{a:"X"})[0]).join(",") === "X"',
    },
    {
      description: 'array vacío',
      assert: 'mapFields([],{a:"X"}).length === 0',
    },
  ]}
