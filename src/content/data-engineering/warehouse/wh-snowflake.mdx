## Snowflake hands-on

**Snowflake** es el warehouse más adoptado en empresas en 2026. A
diferencia de BigQuery (serverless puro), Snowflake usa **virtual
warehouses** (clusters) que tú levantas, escalas y apagas. Esto te da
más control de costos y aislamiento de workloads.

### Estructura

```
account
└── database
    └── schema
        ├── table
        ├── view
        └── stage
```

- **Account**: tu instancia.
- **Database**: agrupamiento alto.
- **Schema**: ~ dataset en BigQuery, namespace en Postgres.
- **Stage**: location intermedia para load/unload (interno o externo a
  S3/GCS).

### Virtual warehouses (compute)

Lo que separa Snowflake del resto: los clusters de compute son
**objetos** que tú creas, levantas y apagas a voluntad.

```sql
CREATE WAREHOUSE etl_wh
  WITH WAREHOUSE_SIZE = 'MEDIUM'
  AUTO_SUSPEND = 60                -- apaga tras 60s sin uso
  AUTO_RESUME = TRUE
  INITIALLY_SUSPENDED = TRUE;

USE WAREHOUSE etl_wh;
```

Tamaños: XS, S, M, L, XL, ... 4XL. Cada salto duplica costo y
paralelismo.

**Clave del costo**: `AUTO_SUSPEND` corto. Si tu pipeline corre 5
min/hora, no tiene sentido tener el warehouse prendido 24/7.

### Multi-cluster warehouses

Para concurrencia alta:

```sql
CREATE WAREHOUSE bi_wh
  WITH WAREHOUSE_SIZE = 'LARGE'
  MIN_CLUSTER_COUNT = 1
  MAX_CLUSTER_COUNT = 5
  SCALING_POLICY = 'STANDARD';
```

Cuando hay cola de queries, Snowflake levanta clusters extra (hasta 5)
y los apaga al desocuparse. Útil para BI con picos.

### Roles y permisos

```sql
USE ROLE ACCOUNTADMIN;

CREATE ROLE analyst;
GRANT USAGE ON WAREHOUSE bi_wh TO ROLE analyst;
GRANT USAGE ON DATABASE analytics TO ROLE analyst;
GRANT USAGE ON SCHEMA analytics.marts TO ROLE analyst;
GRANT SELECT ON ALL TABLES IN SCHEMA analytics.marts TO ROLE analyst;
GRANT ROLE analyst TO USER alice;
```

Snowflake roles son granulares y heredables. Mejor invertir tiempo
desde el día 1 para no terminar con todos siendo `ACCOUNTADMIN`.

### Cargar datos: COPY desde stage

```sql
-- Crear stage externo apuntando a S3
CREATE STAGE my_stage
  URL = 's3://my-bucket/data/'
  STORAGE_INTEGRATION = my_aws_int
  FILE_FORMAT = (TYPE = PARQUET);

-- Copiar a tabla
COPY INTO raw.orders
  FROM @my_stage/orders/
  FILE_FORMAT = (TYPE = PARQUET)
  ON_ERROR = 'CONTINUE'
  PURGE = FALSE;
```

Snowflake paraleliza el COPY entre cores del warehouse — TBs en
minutos. Para datos pequeños, también puedes:

```sql
PUT file://./data.csv @my_stage;            -- sube local a stage
COPY INTO raw.things FROM @my_stage/data.csv;
```

### Time travel: el superpoder

```sql
-- Ver tabla como estaba hace 1 hora
SELECT * FROM orders AT(OFFSET => -60*60);

-- Como estaba en una fecha exacta
SELECT * FROM orders AT(TIMESTAMP => '2026-05-15 10:00:00'::TIMESTAMP);

-- "Antes" de la query que la dañó
SELECT * FROM orders BEFORE(STATEMENT => 'query_id_here');

-- Restaurar
CREATE OR REPLACE TABLE orders AS
  SELECT * FROM orders AT(OFFSET => -60*60);
```

Default retention: 1 día (gratis). Premium: hasta 90 días.

Inválida en BigQuery free tier (max 7 days, configurable). Diferenciador
de Snowflake.

### Zero-copy clone

```sql
CREATE DATABASE analytics_dev CLONE analytics;
```

Clona toda una BD instantáneamente, sin duplicar storage. Ideal para
ambientes dev/staging. Cobras solo storage de los cambios.

### SQL específico de Snowflake

- **`QUALIFY`**: como `WHERE` pero para window functions.
  ```sql
  SELECT * FROM orders
  QUALIFY ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) = 1;
  ```
- **`PIVOT` / `UNPIVOT` nativos** (no como BigQuery con `CASE`).
- **`OBJECT_AGG`, `ARRAY_AGG`**: agregar a JSON/ARRAY.
- **VARIANT**: tipo nativo para JSON semi-estructurado.

### Errores comunes

- **Warehouse XL para queries pequeñas**: factura inflada sin
  beneficio (queries chicas no aprovechan más cores).
- **AUTO_SUSPEND alto** (5 min): pagas tiempo idle.
- **Sin separar warehouses por workload**: BI lento bloquea ETL y
  viceversa.
- **Demasiados ACCOUNTADMIN**: nadie puede revertir cuando alguien
  borra producción por error.
- **No usar zero-copy clone**: pagas storage por copias innecesarias
  para dev.
