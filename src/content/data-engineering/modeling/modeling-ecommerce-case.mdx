## Caso práctico: modelar un e-commerce

Vamos a aplicar todo lo aprendido en un caso realista: modelar el
warehouse analítico de un e-commerce. Vas a tomar las tablas OLTP
(normalizadas, transaccionales) y diseñar el star schema OLAP
correspondiente.

### Lo que tenemos en OLTP (Postgres de la app)

```
customers (id, name, email, country, created_at, updated_at)
products  (id, name, category_id, price, sku, created_at, updated_at)
categories (id, name, department_id)
departments (id, name)
orders    (id, customer_id, status, created_at, total_amount)
order_items (order_id, product_id, quantity, unit_price)
addresses (id, customer_id, line, city, country, is_default)
```

Schema 3FN clásico. Bueno para escribir órdenes; malo para "ventas
mensuales por país y categoría".

### Paso 1: identificar los hechos (facts)

¿Qué eventos numéricos quieres medir? Para un e-commerce, mínimo:

- **Ventas** (orders + order_items): cantidad, monto.
- **Cancelaciones / refunds**.
- **Visitas / sesiones** (si tienes analytics de web).
- **Inventario** (snapshots periódicos).

Empezamos con `fact_sales`. La granularidad más útil: **una fila por
order_item** (línea de orden). Así puedes responder "ventas por
producto" o "ventas por cliente" sin perder detalle.

### Paso 2: identificar dimensiones

Para cada query analítica que prevés, ¿qué necesitas filtrar/agrupar?

- **dim_customer** — quién compró.
- **dim_product** — qué se compró.
- **dim_date** — cuándo.
- **dim_geography** — dónde (de la dirección).

### Paso 4: decisiones clave de diseño

**1. ¿Snowflake o star?** dim_product incluye category y department
denormalizados. **Star**: queries más simples y rápidas. La duplicación
es trivial en motor columnar.

**2. ¿SCD?** dim_customer y dim_product en **Type 2**. Razones:

- Customer puede cambiar segment (paso de "free" a "premium"). Los
  reports históricos deben mostrar el segment al momento de la venta.
- Product cambia precio o categoría. El precio en `dim_product` es el
  "actual"; el precio efectivo de la venta vive en `fact_sales`
  (`unit_price`).

**3. ¿Qué columnas en fact?** Solo:

- FKs a dims (4).
- order_id (degenerate dimension — útil para drill-down).
- quantity, unit_price, line_amount, discount_amount (métricas
  aditivas).
- order_status (para filtros, podría moverse a dim_status pero es
  baja cardinalidad).

**4. dim_date pre-poblada** con 10 años (pasado y futuro): año,
trimestre, mes, día semana, fiscal_year, holiday flag.

### Paso 5: el ETL

```sql
-- fact_sales (incremental por order_id)
INSERT INTO warehouse.fact_sales (
    customer_key, product_key, date_key, geography_key,
    order_id, quantity, unit_price, line_amount, ...
)
SELECT
    dc.customer_key,
    dp.product_key,
    dd.date_key,
    dg.geography_key,
    o.id,
    oi.quantity,
    oi.unit_price,
    oi.quantity * oi.unit_price AS line_amount,
    o.status
FROM source.orders o
JOIN source.order_items oi ON oi.order_id = o.id
JOIN warehouse.dim_customer dc
  ON dc.customer_id = o.customer_id::TEXT
  AND o.created_at BETWEEN dc.effective_from AND COALESCE(dc.effective_to, '9999-12-31')
JOIN warehouse.dim_product dp
  ON dp.product_id = oi.product_id::TEXT
  AND o.created_at BETWEEN dp.effective_from AND COALESCE(dp.effective_to, '9999-12-31')
JOIN warehouse.dim_date dd ON dd.date = o.created_at::DATE
JOIN source.addresses a ON a.customer_id = o.customer_id AND a.is_default = TRUE
JOIN warehouse.dim_geography dg ON dg.country = a.country AND dg.city = a.city
WHERE o.id NOT IN (SELECT order_id FROM warehouse.fact_sales);
```

El JOIN con `BETWEEN effective_from AND effective_to` es **crítico**
en SCD Type 2: te da la versión vigente al momento de la venta.

### Paso 6: queries que el modelo soporta

```sql
-- Top categorías por país en Q1 2026
SELECT g.country, p.category, SUM(f.line_amount) AS revenue
FROM fact_sales f
JOIN dim_geography g USING (geography_key)
JOIN dim_product p USING (product_key)
JOIN dim_date d USING (date_key)
WHERE d.quarter = 1 AND d.year = 2026
GROUP BY 1, 2
ORDER BY revenue DESC;

-- Customer LTV (lifetime value)
SELECT c.customer_id, c.segment, SUM(f.line_amount) AS ltv
FROM fact_sales f
JOIN dim_customer c USING (customer_key)
WHERE c.is_current = TRUE
GROUP BY 1, 2
ORDER BY ltv DESC LIMIT 100;

-- Repeat rate por cohort de signup
SELECT
  date_trunc('month', c.signup_date) AS cohort,
  COUNT(DISTINCT c.customer_id) AS cohort_size,
  COUNT(DISTINCT CASE WHEN f.order_id IS NOT NULL THEN c.customer_id END) AS bought,
  COUNT(DISTINCT f.order_id) / COUNT(DISTINCT c.customer_id)::FLOAT AS orders_per_user
FROM dim_customer c
LEFT JOIN fact_sales f USING (customer_key)
WHERE c.is_current = TRUE
GROUP BY 1;
```

### Errores típicos en este modelado

- **Olvidar la JOIN BETWEEN para SCD2**: terminas usando la dim
  actual aunque la venta fue hace 2 años. Reportes históricos
  inconsistentes.
- **Granularidad por orden** (no por order_item): pierdes drill-down
  por producto.
- **Métricas calculadas en BI**: `line_amount = quantity * unit_price`
  debería estar en la fact, no calculado en cada dashboard.
- **dim_date no pre-poblada**: cada query hace `EXTRACT` en lugar de
  filtrar por columna pre-calculada.
- **Omitir `is_current`**: queries para "estado actual" hacen
  filtrado complejo por effective_to IS NULL.

<Exercise
  id="modeling-ecommerce-exercise"
  title="Calcular LTV"
  description="Implementa `ltv(customers, sales)` que devuelve un objeto { customer_id: total_amount } sumando los amounts de sales por customer_id."
  language="javascript"
  starterCode={`function ltv(customers, sales) {
  return {};
}

const customers = [{id:'A'},{id:'B'},{id:'C'}];
const sales = [
  {customer_id:'A', amount:100},
  {customer_id:'A', amount:50},
  {customer_id:'B', amount:200},
];
console.log(ltv(customers, sales));`}
  solution={`function ltv(customers, sales) {
  const totals = {};
  for (const c of customers) totals[c.id] = 0;
  for (const s of sales) {
    if (totals[s.customer_id] !== undefined) {
      totals[s.customer_id] += s.amount;
    }
  }
  return totals;
}
const customers = [{id:'A'},{id:'B'},{id:'C'}];
const sales = [
  {customer_id:'A', amount:100},
  {customer_id:'A', amount:50},
  {customer_id:'B', amount:200},
];
console.log(ltv(customers, sales));`}
  hints={[
    'Inicializa todos los customers con 0.',
    'Acumula amounts.',
    'Customers sin sales -> 0.',
  ]}
