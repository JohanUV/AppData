## Star schema

El **star schema** es el modelo dimensional más usado en warehouses
analíticos. Lo inventó Ralph Kimball en los 90s y, 30 años después,
sigue siendo el estándar. Si trabajas con BI, dashboards o reports
sobre un warehouse, vas a ver star schemas todos los días.

### La idea

Una tabla central de **hechos** (eventos numéricos: ventas, clicks,
impresiones) rodeada de tablas de **dimensiones** (atributos
descriptivos: clientes, productos, fechas).

### Hechos vs dimensiones

**Tabla de hechos (fact)**:

- Mide algo: cantidades, montos, conteos.
- Crece rápido (millones-billones de filas).
- Mayoría son foreign keys + métricas numéricas.
- Granularidad explícita: "una fila por venta", "una fila por
  click", "una fila por sesión".

**Tablas de dimensiones (dim)**:

- Describen contexto: quién, qué, dónde, cuándo.
- Crecen lento.
- Atributos descriptivos (strings, fechas).
- Surrogate keys (claves sintéticas autogeneradas).

### Por qué este modelo gana

```sql
-- Top productos por país en Q1
SELECT
  c.country,
  p.name AS product,
  SUM(f.amount) AS revenue
FROM fact_sales f
JOIN dim_customer c USING (customer_key)
JOIN dim_product p USING (product_key)
JOIN dim_date d USING (date_key)
WHERE d.quarter = 1 AND d.year = 2026
GROUP BY 1, 2
ORDER BY 1, revenue DESC;
```

3 joins, todos siguiendo el mismo patrón fact↔dim. El optimizador del
motor analítico está hyper-tuneado para este patrón. Comparado con un
schema normalizado (8+ joins), es órdenes de magnitud más rápido.

### Granularidad: la decisión clave

Antes de modelar fact_sales, decide la **grain** (granularidad):

- **Por línea de orden**: una fila por (orden, producto). La más
  detallada — soporta cualquier query.
- **Por orden**: una fila por orden, total agregado. Pierdes
  información de productos individuales.
- **Por día/cliente/producto**: una fila por combinación, ya
  agregada. Lo más resumido — más rápido pero menos flexible.

Regla: empieza con la grain más fina que tu volumen tolere. Vista
agregada se calcula derivada; lo inverso no.

  sale_id      BIGINT,
  customer_key BIGINT REFERENCES dim_customer(customer_key),  -- FK al surrogate
  product_key  BIGINT,
  date_key     INT,
  amount       DECIMAL(12, 2),
  quantity     INT
);`}

### Tipos de hechos

- **Aditivos** (los más amigables): `amount`, `quantity` — se pueden
  sumar en cualquier dimensión.
- **Semi-aditivos**: `inventory_balance` — se suma en tiempo pero no
  por dimensiones (suma de inventario por producto sí; por día no
  tiene sentido, usarías el balance de fin de día).
- **No-aditivos**: `unit_price`, `ratio` — no se suman; promedios o
  weighted averages.

### dim_date: la dimensión que siempre necesitas

```sql
CREATE TABLE dim_date (
  date_key INT PRIMARY KEY,         -- 20260515
  date     DATE,
  year     INT,
  quarter  INT,
  month    INT,
  month_name VARCHAR(20),
  day      INT,
  day_of_week VARCHAR(10),
  is_weekend BOOLEAN,
  is_holiday BOOLEAN,
  fiscal_year INT,
  fiscal_quarter INT
);
```

Pre-poblada con varios años. Tener esto te ahorra `EXTRACT` repetidos
en cada query y soporta lógica de calendario fiscal.

### Errores comunes

- **Decidir grain mal**: si rolas a "por día por país", olvídate de
  reportes por hora.
- **Sin surrogate keys**: cuando el source cambia ids o se migra,
  todo el warehouse rompe.
- **Dimensiones mutables**: cambiar el `country` de un cliente sin
  versionar pierde historia (ver lección de SCD).
- **Hechos no-aditivos sumados ingenuamente**: promedio de
  promedios ≠ promedio total.
- **Demasiadas dim**: 30 dimensiones en una fact = ETL pesadísimo.
  Refactoriza o usa snowflake.
