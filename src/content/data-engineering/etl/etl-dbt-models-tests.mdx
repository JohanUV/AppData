## dbt: models, tests y docs

La lección anterior cubrió los basics de dbt. Ahora vamos a profundidad
en cómo escribir modelos production-grade, tests potentes, y
documentación que se mantiene viva.

### Materializaciones en detalle

```sql
-- Incremental: solo procesa filas nuevas/cambiadas
{{ config(
    materialized='incremental',
    unique_key='order_id',
    on_schema_change='sync_all_columns'
) }}

SELECT *
FROM {{ source('app', 'orders') }}
{% if is_incremental() %}
WHERE updated_at > (SELECT MAX(updated_at) FROM {{ this }})
{% endif %}
```

`is_incremental()` es true solo cuando ya existe la tabla. La primera
vez carga todo; corridas siguientes solo el delta.

`unique_key` permite dedupes (incremental con MERGE en Snowflake/BQ).

### Sources: declarar inputs

```yaml
# models/staging/_sources.yml
version: 2

sources:
  - name: app
    database: raw
    schema: app_postgres
    tables:
      - name: orders
        description: Órdenes del backend
        loaded_at_field: updated_at
        freshness:
          warn_after: {count: 12, period: hour}
          error_after: {count: 24, period: hour}
        columns:
          - name: id
            tests: [unique, not_null]
      - name: customers
```

`freshness` te avisa si los datos crudos no se actualizaron a tiempo
(warning a 12h sin updates, error a 24h). `dbt source freshness`
verifica.

### Tests built-in vs custom

**Built-in (en YAML):**

```yaml
columns:
  - name: order_id
    tests:
      - unique
      - not_null
  - name: amount
    tests:
      - not_null
      - dbt_utils.expression_is_true:
          expression: ">= 0"
  - name: customer_id
    tests:
      - relationships:
          to: ref('dim_customer')
          field: customer_key
```

`dbt_utils` es un paquete con tests extra: rangos, fechas, strings.

**Tests custom (SQL en `tests/`):**

```sql
-- tests/no_orphan_orders.sql
SELECT order_id
FROM {{ ref('fct_revenue_daily') }} f
LEFT JOIN {{ ref('dim_customer') }} c
  ON f.customer_key = c.customer_key
WHERE c.customer_key IS NULL
```

Si la query devuelve **filas**, el test falla. (sin filas = pass)

### Macros: reutilizar SQL

```sql
-- macros/cents_to_dollars.sql
{% macro cents_to_dollars(amount_col) %}
    ({{ amount_col }} / 100)::DECIMAL(12, 2)
{% endmacro %}
```

Uso:

```sql
SELECT
    order_id,
    {{ cents_to_dollars('amount_cents') }} AS amount_dollars
FROM {{ ref('stg_orders') }}
```

Para cálculos complejos repetidos. dbt_utils trae decenas:
`pivot()`, `surrogate_key()`, `date_spine()`.

### Docs auto-generados

```bash
dbt docs generate   # crea catalog.json + manifest.json
dbt docs serve      # sirve en localhost:8080
```

Te da:

- Lineage interactivo (clickeable de modelo a modelo).
- Descripciones de cada columna (de los YAMLs).
- Tests asociados.
- SQL renderizado.
- Linaje column-level (Cloud / dbt-osmosis).

Para que las docs sirvan, **escribir descripciones** en los YAMLs
desde el día 1.

### Patrón staging → marts

```
stg_orders        ←  source app.orders, casts y rename
stg_customers     ←  source app.customers
int_order_summary ←  stg_orders + stg_customers, joins intermedios
fct_revenue_daily ←  int_order_summary, agregaciones finales
dim_customer      ←  stg_customers, SCD desde snapshot
```

Capas evitan que `fct_revenue` referencie `app.orders` directo. Si el
source cambia, solo actualizas `stg_*`.

### Performance tips

- **Incremental** para facts grandes.
- **Tablas en lugar de views** para modelos consultados frecuentemente.
- **Particionamiento + clustering** vía `config(partition_by=..., cluster_by=...)`
  en BigQuery.
- **dbt_utils.deduplicate** para limpieza.
- **Compilar y revisar SQL antes de run**: `dbt compile`.

### Errores comunes

- **YAML sin descripciones**: docs vacías → nadie las lee.
- **Materializar todo como table**: refresh lento.
- **Tests insuficientes**: solo `not_null` no detecta lógica
  rota.
- **No correr `dbt test` en CI**: cambios pasan sin validación.
- **Macros copy-paste sin parametrizar**: pierdes la ventaja de
  abstracción.
- **Snapshots sin schedule**: si los corres una vez al día y la
  fuente se actualiza más rápido, pierdes versiones intermedias.

<Exercise
  id="etl-dbt-models-exercise"
  title="Validador de YAML schema"
  description="Implementa `validateColumnTests(col)` que devuelve true si el objeto column tiene name + al menos 1 test. Ejemplo: {name:'id', tests:['not_null']} → true."
  language="javascript"
  starterCode={`function validateColumnTests(col) {
  return false;
}

console.log(validateColumnTests({name:'id', tests:['not_null','unique']}));   // true
console.log(validateColumnTests({name:'amount'}));                             // false (sin tests)
console.log(validateColumnTests({tests:['not_null']}));                       // false (sin name)`}
  solution={`function validateColumnTests(col) {
  if (!col || typeof col !== 'object') return false;
  if (!col.name) return false;
  return Array.isArray(col.tests) && col.tests.length >= 1;
}
console.log(validateColumnTests({name:'id', tests:['not_null','unique']}));
console.log(validateColumnTests({name:'amount'}));
console.log(validateColumnTests({tests:['not_null']}));`}
  hints={[
    'Verifica name presente.',
    'Verifica tests array con ≥1 elemento.',
  ]}
  tests={[
    {
      description: 'name + tests OK',
      assert: 'validateColumnTests({name:"id",tests:["unique"]}) === true',
    },
    {
      description: 'sin tests',
      assert: 'validateColumnTests({name:"id"}) === false',
    },
    {
      description: 'sin name',
      assert: 'validateColumnTests({tests:["not_null"]}) === false',
