## Operadores y sensores

Un **operador** define **qué hace** una tarea (correr Python,
ejecutar SQL, mover S3 → S3, etc). Un **sensor** es un operador
especial que **espera** a que algo ocurra (un archivo aparezca,
una hora llegue, otro DAG termine).

Conocer los operadores correctos te ahorra escribir Python para todo.

### Operadores built-in más usados

```python
from airflow.operators.python import PythonOperator
from airflow.operators.bash import BashOperator
from airflow.operators.empty import EmptyOperator   # antes "Dummy"

# Para BD/Warehouse
from airflow.providers.postgres.operators.postgres import PostgresOperator
from airflow.providers.snowflake.operators.snowflake import SnowflakeOperator
from airflow.providers.google.cloud.operators.bigquery import BigQueryExecuteQueryOperator

# Para nubes
from airflow.providers.amazon.aws.operators.s3 import S3CreateObjectOperator
from airflow.providers.amazon.aws.transfers.s3_to_redshift import S3ToRedshiftOperator
```

### PythonOperator: el comodín

```python
def my_task(**context):
    print('processing...')
    return 'done'

task = PythonOperator(
    task_id='my_task',
    python_callable=my_task,
    op_kwargs={'param1': 'value'},   # se inyectan como kwargs
    op_args=['positional1'],
)
```

Ojo con tareas pesadas: el worker la ejecuta. Para procesamiento real,
delegar a un sistema (BigQuery, Spark) y solo orquestar desde Airflow.

### BashOperator y SQL operators

```python
extract = BashOperator(
    task_id='extract',
    bash_command='python /opt/scripts/extract.py --date {{ ds }}',
)

transform = PostgresOperator(
    task_id='transform',
    postgres_conn_id='warehouse_pg',
    sql='/sql/transform.sql',         # path relativo al DAGs folder
    parameters={'date': '{{ ds }}'},
)
```

`postgres_conn_id` apunta a una **Connection** definida en la UI de
Airflow (Admin → Connections). No hardcodees credenciales en el DAG.

### Sensores

Un sensor "duerme" hasta que una condición se cumple. Útil para esperar
archivos, particiones, horas externas.

```python
from airflow.providers.amazon.aws.sensors.s3 import S3KeySensor
from airflow.sensors.time_sensor import TimeSensor
from airflow.sensors.external_task import ExternalTaskSensor

wait_for_file = S3KeySensor(
    task_id='wait_for_file',
    bucket_name='raw-data',
    bucket_key='orders/{{ ds }}/data.parquet',
    poke_interval=60,                # cada 60s
    timeout=60 * 60 * 4,              # max 4 horas
    mode='reschedule',                # crítico (ver abajo)
)

wait_for_upstream_dag = ExternalTaskSensor(
    task_id='wait_for_extract_dag',
    external_dag_id='extract_orders',
    external_task_id='final_task',
    mode='reschedule',
)
```

### Modes: poke vs reschedule

- **`poke`** (default): la tarea ocupa un slot del worker mientras
  espera. Si esperas 4 horas, gastas un slot todo ese tiempo.
- **`reschedule`**: el sensor revisa, si no cumple suelta el slot y
  vuelve a programarse en `poke_interval`.

Para esperas largas (>5 minutos), **siempre `reschedule`**. Sino te
quedas sin slots y todo el cluster se traba.

### Evita el anti-patrón: muchos sensores en cascada

Si tu DAG tiene 5 sensores en serie esperando 4 horas cada uno,
estás haciendo algo mal. Considera:

- **Trigger basado en eventos** (Lambda → API call para trigger DAG).
- **Datasets de Airflow** (Airflow 2.4+): producer/consumer pattern,
  un DAG dispara otro al producir datos.

```python
from airflow.datasets import Dataset

orders_dataset = Dataset('s3://raw/orders/')

# DAG productor
@task(outlets=[orders_dataset])
def extract():
    ...

# DAG consumidor
@dag(schedule=[orders_dataset], ...)
def transform_pipeline():
    ...
```

### Operadores que vale la pena conocer

- `EmptyOperator`: placeholder, no hace nada (útil como "barrier").
- `BranchPythonOperator`: ejecuta una de varias ramas según lógica.
- `KubernetesPodOperator`: ejecuta un container en K8s.
- `EmailOperator`: manda email.
- `TriggerDagRunOperator`: dispara otro DAG.
- `*ToS3*Operator` y `S3To*Operator`: cientos de transfers para
  todos los stacks.

Más en [Airflow Providers Index](https://airflow.apache.org/docs/).

### Errores comunes

- **PythonOperator con código que mejor delega a otro sistema**: si
  hace 10 GB de transformación, sufre el worker. Delega a Spark/BigQuery.
- **Sensor en mode poke con timeout largo**: bloqueo de slots.
- **No usar Connections**: credenciales en el código → leak.
- **Templates Jinja en argumentos no template-able**: solo ciertos
  fields aceptan templates (mira la doc del operator).
- **Mezclar TaskFlow y operators clásicos sin razón**: confunde al
  lector.
