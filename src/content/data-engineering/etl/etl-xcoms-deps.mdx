## XComs y dependencias

**XCom** (cross-communication) es el mecanismo de Airflow para pasar
**datos pequeños** entre tareas. Esta lección cubre cómo usarlos
correctamente, qué NO meterles, y cómo expresar dependencias
complejas (paralelo, ramas condicionales, fan-in/fan-out).

### Cómo funciona XCom

Cada vez que un PythonOperator (o @task) retorna un valor, Airflow lo
serializa con pickle y lo guarda en la **metadata DB**. Otra tarea lo
lee con `xcom_pull`.

```python
def extract(**context):
    return [{'id': 1}, {'id': 2}]    # va a XCom

def transform(**context):
    ti = context['ti']
    records = ti.xcom_pull(task_ids='extract')
    return [r for r in records if r['id'] > 0]
```

Con TaskFlow API, lo mismo es transparente:

```python
@task
def extract():
    return [{'id': 1}, {'id': 2}]

@task
def transform(records):    # records viene del XCom de extract
    return [r for r in records if r['id'] > 0]

records = extract()
transform(records)
```

### Qué NO meter en XCom

- **Datasets grandes**. Como Airflow lo serializa a la metadata DB,
  meter 1GB de DataFrame es disastre. Regla: < 48KB (default size
  limit en algunos backends).
- **Objetos no serializables**. Conexiones, file handles, modelos ML.
- **Datos sensibles**. Quedan en BD; cualquiera con acceso los lee.

Para datasets grandes: persiste en S3/GCS y pasa por XCom **el path**:

```python
@task
def extract():
    df = pd.read_sql(...)
    path = f's3://staging/orders-{{ ds }}.parquet'
    df.to_parquet(path)
    return path                      # solo el path en XCom

@task
def transform(path):
    df = pd.read_parquet(path)
    ...
```

### XCom backends custom

Para datasets más grandes sin reescribir todo, configuras un **custom
backend** que serializa a S3 transparentemente:

```python
# airflow.cfg
xcom_backend = airflow.providers.amazon.aws.xcoms.s3_xcom_backend.S3XComBackend
```

XCom debajo guarda en S3; el código sigue usando `xcom_pull` /
returns como siempre.

### Dependencias complejas

**Paralelo (fan-out):**

```python
extract = PythonOperator(...)
clean_a = PythonOperator(...)
clean_b = PythonOperator(...)
clean_c = PythonOperator(...)
load = PythonOperator(...)

extract >> [clean_a, clean_b, clean_c] >> load
```

`>>` con lista significa "después de extract, las 3 cleans en paralelo,
y cuando todas terminan, load".

**Fan-in:**

```python
[extract_stripe, extract_shopify, extract_paypal] >> consolidate >> load
```

**TaskFlow equivalente:**

```python
@task
def extract(source): ...

stripe = extract('stripe')
shopify = extract('shopify')
paypal = extract('paypal')

@task
def consolidate(s, sh, p): ...

consolidate(stripe, shopify, paypal)
```

### Branching: ejecutar una rama según lógica

```python
from airflow.operators.python import BranchPythonOperator

def choose_path(**context):
    ds = context['ds']
    if ds.endswith('-01'):
        return 'monthly_full_load'
    return 'daily_incremental'

branch = BranchPythonOperator(
    task_id='branch',
    python_callable=choose_path,
)

monthly = PythonOperator(task_id='monthly_full_load', ...)
daily = PythonOperator(task_id='daily_incremental', ...)
join = EmptyOperator(task_id='join', trigger_rule='none_failed_min_one_success')

branch >> [monthly, daily] >> join
```

`trigger_rule` ajusta cuándo `join` se ejecuta. Default es
`all_success`, pero como una rama va a ser skipped, necesitas
`none_failed_min_one_success`.

### Trigger rules útiles

- `all_success` (default): todas las upstream OK.
- `all_failed`: todas las upstream fallaron.
- `one_success`: al menos una OK (resto puede haber fallado).
- `none_failed`: ninguna falló (skipped OK).
- `none_failed_min_one_success`: ninguna falló y al menos una corrió.
- `always`: corre sin importar nada.

### Errores comunes

- **XCom con DataFrames grandes**: BD se hincha, scheduler lento.
- **Olvidar trigger_rule en branching**: la rama no-tomada queda
  skipped, downstream falla en cadena.
- **Dependencias circulares accidentales**: Airflow las detecta y
  rechaza el DAG.
- **Confiar en orden de ejecución de tareas paralelas**: nunca lo
  asumas; el executor decide.
- **No documentar XComs**: el siguiente que toque el DAG no sabe qué
  espera cada tarea.
