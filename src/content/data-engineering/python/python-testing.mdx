## Testing de pipelines con pytest

Un pipeline sin tests es una bomba de tiempo. Los datos cambian; las
APIs cambian; tu refactor de "una línea inocente" rompe el dashboard
del CFO. Tests unitarios + integración son la diferencia entre un
pipeline que funciona los lunes y uno que funciona siempre.

### pytest en 60 segundos

```python
# test_transforms.py
def add(a, b):
    return a + b

def test_add():
    assert add(2, 3) == 5

def test_add_negative():
    assert add(-1, 1) == 0
```

```bash
pytest                  # corre todo
pytest test_xx.py        # un archivo
pytest -k "negative"     # solo tests cuyo nombre matchea
pytest -v                # verbose
pytest --cov=src         # con cobertura
```

Convención: archivos `test_*.py`, funciones `def test_*`. pytest los
descubre automáticamente.

### Parametrize: muchos casos sin repetir

```python
import pytest

@pytest.mark.parametrize('amount,bucket', [
    (5, 'small'),
    (50, 'medium'),
    (500, 'large'),
    (5000, 'huge'),
])
def test_bucketize(amount, bucket):
    assert bucketize(amount) == bucket
```

Un solo test definicional, 4 casos. Si uno falla, pytest te dice
exactamente cuál.

### Mocks para llamadas externas

Los pipelines llaman APIs, BDs, S3. En tests no quieres tocar
producción:

```python
from unittest.mock import patch, Mock

def fetch_orders(api_url):
    return requests.get(api_url).json()

def test_fetch_orders():
    with patch('requests.get') as mock_get:
        mock_get.return_value.json.return_value = [{'id': 1}]
        result = fetch_orders('http://api')
        assert result == [{'id': 1}]
        mock_get.assert_called_once_with('http://api')
```

`patch` reemplaza `requests.get` con un mock dentro del bloque. Puedes
verificar qué se llamó y con qué args.

### Tests de DataFrames

Pandas trae helpers:

```python
import pandas as pd
from pandas.testing import assert_frame_equal

def test_transform():
    inp = pd.DataFrame({'a': [1, 2]})
    expected = pd.DataFrame({'a': [1, 2], 'a_doubled': [2, 4]})
    out = transform(inp)
    assert_frame_equal(out, expected)
```

`assert_frame_equal` compara columnas, tipos y valores. Te da un diff
útil cuando no coinciden.

### Test pyramide para pipelines

<Diagram type="mermaid">
{`flowchart TB
  A[Tests E2E<br/>pocos, lentos<br/>pipeline completo<br/>con fixtures]
  B[Tests de integración<br/>algunos<br/>componentes hablando<br/>BD/API reales]
  C[Tests unitarios<br/>muchos, rápidos<br/>funciones puras]
  C --> B --> A`}
</Diagram>

- **Unitarios** (80%): funciones de transformación puras (map de
  columnas, parsers). Sub-segundo, sin red ni disco.
- **Integración** (15%): código que toca BD/API/storage. Usa BD local
  (Postgres en docker-compose) o LocalStack para AWS.
- **E2E** (5%): correr el pipeline completo con datos sintéticos.

### Tests de calidad de datos: Great Expectations

Para validar **datos** (no código), librerías especializadas:

```python
import great_expectations as ge

df_ge = ge.from_pandas(df)
df_ge.expect_column_to_exist('user_id')
df_ge.expect_column_values_to_not_be_null('user_id')
df_ge.expect_column_values_to_be_between('amount', min_value=0, max_value=10_000)
```

Lo corres en producción como un step más del pipeline: si los datos
violan expectativas, el pipeline falla y suena la alarma.

### Errores comunes

- **No tener tests**. La excusa "es un pipeline, no un app" no vale.
- **Solo tests E2E**: cuando algo falla, no sabes qué fue. Combina
  niveles.
- **Tests que tocan producción**. Aislalos con mocks o test envs.
- **`assert df.shape == (10, 5)`** sin chequear contenido. Forma
  correcta + datos basura = test verde con datos malos.
- **Mocks que mienten**: validar que `mock.return_value` refleje
  realmente lo que la API devuelve. Si la API cambia, tu test no
  detecta el cambio.

<Exercise
  id="python-testing-exercise"
  title="Mock simple"
  description="Implementa `withMock(originalFn, mockFn, callback)` que ejecuta callback con originalFn temporalmente reemplazado por mockFn, y restaura al final. Devuelve el resultado de callback."
  language="javascript"
  starterCode={`function withMock(target, key, mockFn, callback) {
  return null;
}

const obj = { fetch: () => 'real' };
const result = withMock(obj, 'fetch', () => 'mocked', () => obj.fetch());
console.log(result);  // 'mocked'
console.log(obj.fetch());  // 'real' — restaurado`}
  solution={`function withMock(target, key, mockFn, callback) {
  const original = target[key];
  target[key] = mockFn;
  try {
    return callback();
  } finally {
    target[key] = original;
  }
}
const obj = { fetch: () => 'real' };
const result = withMock(obj, 'fetch', () => 'mocked', () => obj.fetch());
console.log(result);
console.log(obj.fetch());`}
  hints={[
    'Guarda el original, reemplaza, ejecuta, restaura.',
    'try/finally para que el restore corra incluso con error.',
  ]}
  tests={[
    {
      description: 'el mock se ejecuta',
      assert: 'withMock({f:()=>1},"f",()=>2,(o)=>1) === 1',
    },
    {
      description: 'restaura el original',
      assert: '(()=>{const o={f:()=>1}; withMock(o,"f",()=>2,()=>{}); return o.f()})() === 1',
    },
    {
      description: 'restaura incluso con error',
      assert: '(()=>{const o={f:()=>1}; try{withMock(o,"f",()=>2,()=>{throw new Error()})}catch{} return o.f()})() === 1',
