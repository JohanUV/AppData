## SQLAlchemy y conexión a BDs

**SQLAlchemy** es la librería de Python para conectarse a cualquier
base de datos relacional. Soporta Postgres, MySQL, SQLite, SQL Server,
Oracle, BigQuery, Snowflake — todo con la misma API. Para Data
Engineering, suele ser la capa de transporte entre tu pipeline y los
warehouses.

### Engine y connection

```python
from sqlalchemy import create_engine, text

engine = create_engine(
    'postgresql+psycopg2://user:pass@host:5432/db',
    pool_size=5,
    pool_pre_ping=True,
)

with engine.connect() as conn:
    result = conn.execute(text('SELECT 1'))
    print(result.scalar())
```

Cada motor tiene su URL: `postgresql://`, `mysql://`, `sqlite:///`,
`bigquery://project`, etc. La parte `+driver` indica qué driver Python
usar (psycopg2 para Postgres, mysqlconnector para MySQL).

`pool_pre_ping=True` chequea conexiones antes de usarlas — evita
errores "MySQL server has gone away" en pipelines que viven horas.

### Lectura: usa Pandas

```python
import pandas as pd

df = pd.read_sql(
    'SELECT * FROM payments WHERE created_at >= %(since)s',
    engine,
    params={'since': '2026-01-01'},
)
```

`pd.read_sql` con engine de SQLAlchemy es la forma idiomática de cargar
queries a DataFrames. **Siempre usa parámetros**, nunca `f-strings` con
input externo: SQL injection.

### Escritura: insert / upsert

```python
df.to_sql(
    'payments_staging',
    engine,
    if_exists='append',     # 'fail', 'replace', 'append'
    index=False,
    chunksize=10_000,
    method='multi',         # acelera muchísimo en Postgres
)
```

<Callout type="warning" title="`if_exists='replace'` borra la tabla">
  No solo trunca, dropea. Pierdes índices, constraints, permisos. Es
  para staging temporal, no para producción.
</Callout>

Para upsert (INSERT ... ON CONFLICT), SQLAlchemy 2.0 te da:

```python
from sqlalchemy.dialects.postgresql import insert

stmt = insert(payments_table).values(rows)
stmt = stmt.on_conflict_do_update(
    index_elements=['id'],
    set_={'amount': stmt.excluded.amount},
)
with engine.begin() as conn:
    conn.execute(stmt)
```

### Transacciones

```python
with engine.begin() as conn:    # auto-commit on success, rollback on error
    conn.execute(text('UPDATE accounts SET balance = balance - 100 WHERE id = 1'))
    conn.execute(text('UPDATE accounts SET balance = balance + 100 WHERE id = 2'))
# si una falla, las dos hacen rollback
```

`engine.begin()` te da una transacción explícita. `engine.connect()` no:
debes commitear manualmente.

### El Core API vs el ORM

SQLAlchemy tiene dos niveles:

- **Core**: SQL imperativo con builder Python. Lo que vimos arriba.
- **ORM**: clases que mapean tablas. `User.query.filter_by(...)`.

Para Data Engineering, el **Core** es lo correcto. ORMs brillan en
aplicaciones web; en pipelines son innecesario overhead. Si quieres
type-safety y abstracción, considera **dbt** (lección de ETL) en lugar
del ORM de SQLAlchemy.

### Patrón típico en un pipeline

```python
def extract(engine):
    return pd.read_sql(
        '''SELECT * FROM payments
           WHERE created_at >= :since AND created_at < :until''',
        engine,
        params={'since': start, 'until': end},
    )

def transform(df):
    df['amount_eur'] = df['amount'] * 0.92
    return df

def load(df, engine):
    df.to_sql('payments_clean', engine, if_exists='append',
              index=False, chunksize=10_000, method='multi')
```

### Errores comunes

- **f-string con input externo**: SQL injection. Siempre `text(':name')`
  + params dict.
- **No cerrar conexiones**: `engine.connect()` SIN `with` te deja
  conexiones abiertas hasta agotar el pool.
- **`if_exists='replace'`** en producción: dropea la tabla.
- **`pool_size` enorme** "por las dudas": cada conexión consume RAM en
  el motor. Postgres con 1000 conexiones se cae.
- **Sin retries**: la conexión a la BD se cae 1-2 veces al día. Tus
  pipelines tienen que ser resilientes.
