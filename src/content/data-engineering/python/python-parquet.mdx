## Parquet y formatos columnares

CSV es el formato que todo el mundo entiende. También es el peor para
analítica: lento de leer, pesado en disco, sin tipos. **Parquet** es lo
que usa la industria para datasets analíticos. Si haces un pipeline que
lee/escribe en S3, GCS o un data lake, vas a usar Parquet.

### Row-oriented vs columnar

CSV es **row-oriented**: cada línea es una fila, todos los campos
juntos.

```
user_id,country,amount
1,CO,100
2,MX,50
3,CO,200
```

Para una query "suma de amount" tienes que leer **todo el archivo** —
incluyendo country aunque no lo uses.

Parquet es **columnar**: las columnas se almacenan separadas y
comprimidas independientemente.

```
[user_id: 1, 2, 3]
[country: CO, MX, CO]
[amount: 100, 50, 200]
```

Para "suma de amount" lees solo esa columna. Resultado: 5-50× menos I/O.

### Lectura y escritura

```python
import pandas as pd
import pyarrow as pa
import pyarrow.parquet as pq

# Pandas (más fácil)
df = pd.read_parquet('payments.parquet')
df.to_parquet('out.parquet', compression='snappy')

# PyArrow (más control)
table = pq.read_table('payments.parquet', columns=['user_id', 'amount'])
df = table.to_pandas()
```

`columns=` en PyArrow lee SOLO esas columnas — escala a archivos de
GB sin cargar todo.

### Tipos preservados

Parquet guarda el schema con los datos. CSV no — todo es string hasta
que tu lector adivina:

```python
df_csv = pd.read_csv('data.csv')
df_csv['user_id'].dtype          # object (string!) si tiene leading zeros
df_csv['date'].dtype             # object — no parseo automático

df_pq = pd.read_parquet('data.parquet')
df_pq['user_id'].dtype           # int64 — el tipo correcto
df_pq['date'].dtype              # datetime64[ns]
```

Esto **previene bugs** silenciosos en pipelines.

### Compresión

```python
df.to_parquet('out.parquet', compression='snappy')   # rápido, default
df.to_parquet('out.parquet', compression='gzip')     # más pequeño, lento
df.to_parquet('out.parquet', compression='zstd')     # equilibrio moderno
```

Snappy es la default — buena para hot data. Zstd da mejor compresión
con CPU similar; cada vez más usado.

Particionar bien depende del patrón de query. Regla: particiona por la
columna que más usas en `WHERE`. Para datos de eventos: por fecha
(`date` o `year/month/day`).

### Otros formatos columnares

- **ORC**: similar a Parquet, optimizado para Hive. Verás en stacks
  Hadoop legacy.
- **Avro**: row-oriented pero con schema. Bueno para mensajes Kafka,
  malo para analytics.
- **Iceberg / Delta / Hudi**: "table formats" sobre Parquet. Te dan
  ACID, time travel, schema evolution.

Para Data Engineering moderno: Parquet por default; Iceberg/Delta para
lakehouse.

### Limites de Parquet

- **No para mutaciones**: append es OK, update/delete no es trivial
  (eso es lo que aporta Iceberg/Delta).
- **No streaming pequeño**: archivos diminutos crean overhead
  (problema clásico "small files"). Idealmente cada Parquet = 100MB-1GB.
- **No editable a mano**: binario.

### Errores comunes

- **Generar millones de archivos pequeños** (1KB cada uno) en S3.
  Compactar regularmente. Reglas: archivos `<=` 128MB son small.
- **No usar `columns=`** para columns selection: cargas data inútil.
- **Particionar por columnas con altísima cardinalidad** (timestamp
  exacto, user_id). Genera explosión de directorios. Prefiere truncar
  a día.
- **CSV cuando podrías Parquet**: 5× más espacio, 10× más lento, sin
  tipos. Salvo para handoff con humanos, evítalo.
