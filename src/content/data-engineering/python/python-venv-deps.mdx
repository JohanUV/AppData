## Entornos virtuales y dependencias

Python tiene un problema histórico: si instalas paquetes globalmente,
chocan entre proyectos. El proyecto A necesita pandas 1.5; el B
necesita 2.1. Sin aislamiento, el último que instalas pisa al
anterior. **Entornos virtuales** son la solución.

### venv: lo built-in

```bash
# Crear entorno
python -m venv .venv

# Activar
source .venv/bin/activate         # Linux/macOS
.venv\Scripts\activate             # Windows

# Instalar paquetes (van solo al .venv)
pip install pandas requests

# Salir
deactivate
```

`.venv/` es una carpeta con un Python aislado. Cuando lo activas, `pip
install` y `python` apuntan ahí, no al sistema.

**Siempre añade `.venv/` al `.gitignore`**. No se versiona.

### requirements.txt: declarando dependencias

```bash
# Generar lista de lo instalado actualmente
pip freeze > requirements.txt

# Instalar lo declarado en otra máquina
pip install -r requirements.txt
```

`pip freeze` te da algo así:

```
pandas==2.1.3
numpy==1.26.0
requests==2.31.0
```

Estos pins exactos son **buenos para reproducibilidad** (mismo entorno
en dev/CI/prod) pero malos para desarrollo (cualquier upgrade requiere
manual). Patrón común: `requirements.in` con rangos sueltos +
`requirements.txt` generado vía `pip-compile` (de pip-tools).

### Poetry: gestión moderna

Poetry maneja dependencias + virtualenv + packaging con un comando:

```bash
# Inicializar proyecto
poetry init

# Agregar paquete (instala + actualiza pyproject.toml + lock)
poetry add pandas
poetry add --group dev pytest

# Instalar lo declarado
poetry install

# Activar shell del entorno
poetry shell
```

`pyproject.toml` reemplaza `setup.py` y `requirements.txt`:

```toml
[tool.poetry]
name = "my-pipeline"
version = "0.1.0"

[tool.poetry.dependencies]
python = "^3.11"
pandas = "^2.1"
requests = "^2.31"

[tool.poetry.group.dev.dependencies]
pytest = "^7.4"
```

`poetry.lock` (auto-generado, se commitea) congela versiones exactas
para todos. El `^2.1` significa "compatible con 2.1, ≤3.0".

### uv: el nuevo rápido

[`uv`](https://github.com/astral-sh/uv) es un instalador 10-100× más
rápido que pip, escrito en Rust:

```bash
uv venv                       # crea .venv
uv pip install pandas         # instala
uv pip compile requirements.in -o requirements.txt
```

Cada vez más adoptado en CI por velocidad.

### conda y mamba

Para data science con dependencias nativas pesadas (NumPy con MKL,
GeoPandas con GDAL), conda maneja cosas que pip no puede (binarios C++).
Mamba es conda con un solver más rápido.

```bash
mamba create -n my-pipeline python=3.11 pandas pyarrow -c conda-forge
mamba activate my-pipeline
```

En Data Engineering puro: pip + Poetry/uv suelen bastar. Conda gana
cuando trabajas con ciencia/ML pesada.

### Estructura típica de un pipeline

```
my-pipeline/
├── .venv/               (gitignored)
├── .python-version      (3.11.7)
├── pyproject.toml       (deps + metadata)
├── poetry.lock          (versions exactas)
├── README.md
├── src/
│   └── pipeline/
│       ├── __init__.py
│       ├── extract.py
│       ├── transform.py
│       └── load.py
├── tests/
│   └── test_transform.py
└── .env.example         (template, sin secretos)
```

### Errores comunes

- **Instalar paquetes globalmente**. Saturan tu sistema, chocan entre
  proyectos.
- **Versionar `.venv/`**. Pesa MB, contiene paths absolutos, no es
  portable.
- **No fijar versión de Python**. Tu colega usa 3.9, tú 3.12, algo
  rompe sin que sepan por qué.
- **`pip install` sin `requirements.txt` o lock**. Dos máquinas, dos
  estados distintos.
- **Mezclar pip y conda**. Romper depencies en formas creativas.
  Decide uno y stick to it.

## Ejercicio práctico

<Exercise
  id="python-venv-exercise"
  title="Parser de requirements.txt"
  description="Implementa `parseRequirements(text)` que parse un requirements.txt simple y devuelva un objeto { paquete: version }. Líneas que empiezan con # son comentarios."
  language="javascript"
  starterCode={`function parseRequirements(text) {
  return {};
}

const text = \`# data
pandas==2.1.3
numpy==1.26.0
# http
requests==2.31.0\`;
console.log(parseRequirements(text));`}
  solution={`function parseRequirements(text) {
  const out = {};
  for (const line of text.split('\\n')) {
    const trimmed = line.trim();
    if (!trimmed || trimmed.startsWith('#')) continue;
    const m = trimmed.match(/^([a-zA-Z0-9_-]+)==(.+)$/);
    if (m) out[m[1]] = m[2];
  }
  return out;
}
const text = \`# data
pandas==2.1.3
numpy==1.26.0
# http
requests==2.31.0\`;
console.log(parseRequirements(text));`}
  hints={[
    'Split por newline.',
    'Skip lineas vacías y comentarios.',
    'Regex para "name==version".',
  ]}
  tests={[
    {

      assert: 'parseRequirements("pandas==2.1.3").pandas === "2.1.3"',
    },
    {

    },
    {
