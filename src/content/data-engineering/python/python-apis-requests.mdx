## Consumo de APIs con requests

Buena parte de los datos en producción no vienen de archivos: vienen
de **APIs**. Stripe, Salesforce, Shopify, Slack — todos exponen REST
endpoints que tu pipeline tiene que consultar regularmente. La librería
estándar de facto en Python para esto es **requests**.

### Una llamada GET básica

```python
import requests

resp = requests.get('https://api.github.com/users/octocat')
resp.raise_for_status()         # error si status >= 400
data = resp.json()
print(data['name'], data['public_repos'])
```

`raise_for_status()` es **importante**: por defecto requests no falla
con HTTP 4xx/5xx, te devuelve el response. Si no chequeas, asumes que
todo bien y procesas data inválida.

### Headers y query params

```python
resp = requests.get(
    'https://api.example.com/v1/orders',
    headers={'Authorization': 'Bearer ' + token},
    params={'status': 'paid', 'limit': 100},
    timeout=30,                  # ¡siempre!
)
```

**Siempre** pon `timeout`. Sin él, una API lenta puede colgar tu
pipeline indefinidamente.

### POST con JSON

```python
resp = requests.post(
    'https://api.example.com/v1/webhooks',
    json={'url': 'https://my.site', 'event': 'order.created'},
    headers={'Authorization': 'Bearer ' + token},
    timeout=30,
)
```

Pasar `json=` (no `data=`) auto-serializa y agrega el header
`Content-Type: application/json`.

### Paginación

Casi toda API real pagina. Patrones comunes:

```python
# Estilo offset/limit
all_items = []
offset = 0
while True:
    resp = requests.get(url, params={'offset': offset, 'limit': 100}, timeout=30)
    resp.raise_for_status()
    page = resp.json()['items']
    if not page:
        break
    all_items.extend(page)
    offset += 100

# Estilo cursor (mejor)
cursor = None
while True:
    params = {'limit': 100}
    if cursor:
        params['cursor'] = cursor
    resp = requests.get(url, params=params, timeout=30).json()
    all_items.extend(resp['items'])
    cursor = resp.get('next_cursor')
    if not cursor:
        break
```

Cursor pagination escala mejor — no salta filas si el dataset cambia
durante la iteración.

### Rate limiting cliente-side

Algunas APIs son estrictas (60 req/min). Mejor no llegar al 429:

```python
import time

def rate_limited_get(urls, calls_per_second=2):
    interval = 1.0 / calls_per_second
    last = 0
    for url in urls:
        wait = interval - (time.time() - last)
        if wait > 0:
            time.sleep(wait)
        yield requests.get(url, timeout=30).json()
        last = time.time()
```

### Async para paralelismo

`requests` es síncrono. Para hacer 100 calls en paralelo usa `aiohttp`
o `httpx` con asyncio:

```python
import httpx, asyncio

async def fetch(client, url):
    r = await client.get(url, timeout=30)
    return r.json()

async def main(urls):
    async with httpx.AsyncClient() as client:
        return await asyncio.gather(*[fetch(client, u) for u in urls])

results = asyncio.run(main(urls))
```

10× más rápido para 100 endpoints lentos.

### Errores comunes

- **No poner `timeout=`**. Tu pipeline se cuelga indefinidamente.
- **No chequear `raise_for_status()`** y procesar respuestas con error.
- **Reintentar 4xx**. Son errores tuyos, no del servidor; reintento
  inútil y a veces destructivo.
- **No pagar tokens fuera del repo**: ya lo viste en Git, repítalo
  aquí: tokens en `.env`, jamás hardcodeados.
- **No respetar rate limits**: te bloquean la API key.
- **Cargar el JSON entero en memoria** cuando es enorme: usa
  `stream=True` y procesa por chunks.
