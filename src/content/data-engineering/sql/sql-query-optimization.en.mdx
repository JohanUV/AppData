## Query optimization

A slow query is never magic. There's a concrete bottleneck: full scan,
missing index, JOIN exploding rows, or planner misestimating
cardinalities. This lesson is the diagnostic playbook.

### Diagnostic process

<Diagram type="mermaid">
{`flowchart TD
  A[Slow query] --> B[Reproduce<br/>with EXPLAIN ANALYZE]
  B --> C{Where is<br/>the time?}
  C -->|Seq Scan| D[Missing index<br/>or unselective filter]
  C -->|Sort| E[Expensive ORDER BY<br/>+ no index]
  C -->|Huge Hash Join| F[Intermediate table<br/>inflates — review JOINs]
  C -->|Nested Loop with high N| G[Correlated subquery<br/>or repeated lookup]
  D --> H[Create proper index]
  E --> H
  F --> I[Filter early,<br/>JOIN small first]
  G --> J[Rewrite as JOIN]`}
</Diagram>

### Read EXPLAIN like a detective

Start with the **most expensive operation**: highest time or cost.
Inside-out.

```
QUERY PLAN
Sort  (cost=2000.00..2050.00 rows=20000 width=64)
  Sort Key: created_at DESC
  ->  Hash Join  (cost=500.00..1500.00 rows=20000 width=64)
        Hash Cond: (o.user_id = u.id)
        ->  Seq Scan on orders o  (cost=0.00..800.00 rows=80000)
              Filter: (status = 'paid')
        ->  Hash  (cost=200.00..200.00 rows=10000)
              ->  Seq Scan on users u  (cost=0.00..200.00 rows=10000)
                    Filter: (country = 'CO')
```

Diagnosis: Seq Scan on `orders` with `status='paid'` filter. If selective,
**missing index** on status (or compound with user_id).

### Strategy 1: filter early

Apply WHERE as soon as possible. Filter **before** joining, not after:

```sql
-- ❌ Slow: join everything, filter at the end
SELECT *
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.country = 'CO' AND o.status = 'paid';

-- ✓ Better: pre-filter each side
WITH co_users AS (
  SELECT * FROM users WHERE country = 'CO'
),
paid_orders AS (
  SELECT * FROM orders WHERE status = 'paid'
)
SELECT *
FROM co_users u
LEFT JOIN paid_orders o ON u.id = o.user_id;
```

### Strategy 2: indexes on right columns

`WHERE`, `JOIN ON`, `ORDER BY` with LIMIT are candidates. Three rules:

1. Filtered often (high cardinality).
2. Selective (small fraction matches).
3. Frequently queried.

For `WHERE col1 = X AND col2 > Y ORDER BY col3`, consider compound index
`(col1, col2, col3)`.

### Strategy 3: avoid SELECT *

`SELECT *` reads every column. Heavy TEXT/BLOB you didn't need? You drag
it. With covering index, **index-only scan** doesn't even touch the
table.

```sql
CREATE INDEX idx_users_email_id ON users(email, id);
SELECT id FROM users WHERE email = 'x';
```

### Strategy 4: LIMIT changes the plan

```sql
SELECT * FROM events ORDER BY created_at DESC;          -- sorts everything
SELECT * FROM events ORDER BY created_at DESC LIMIT 100; -- index scan, stop at 100
```

The optimizer is more aggressive with small LIMIT.

### Strategy 5: expensive aggregations

`COUNT(DISTINCT col)` on huge tables is costly. Alternatives:

- `APPROX_COUNT_DISTINCT(col)` (BigQuery, Redshift): under 1% error, much
  faster.
- Pre-aggregate to a daily table.

### Strategy 6: partitioning (warehouses)

In BigQuery/Snowflake/Redshift no B-trees. Optimization via:

- **Partition** by date or typical filter column.
- **Cluster** by frequent WHERE columns.
- **Materialized aggregations**.

`WHERE date = '2026-01-15'` on a date-partitioned table only scans that
partition — orders of magnitude faster.

### Strategy 7: avoid accidental cartesians

```sql
SELECT *
FROM orders, order_items;   -- implicit cross join!
```

N × M rows. Always explicit `JOIN ... ON`.

### Strategy 8: refactor correlated subqueries

```sql
-- ❌ N evaluations
SELECT u.name,
  (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS n
FROM users u;

-- ✓ Single pass
SELECT u.name, COALESCE(o.n, 0) AS n
FROM users u
LEFT JOIN (
  SELECT user_id, COUNT(*) AS n FROM orders GROUP BY user_id
) o ON u.id = o.user_id;
```

### Strategy 9: keep stats fresh

Postgres uses statistics (`ANALYZE`) for planning. If tables grow without
analyze, plans degrade. Schedule `VACUUM ANALYZE`.

### Common mistakes

- **Optimizing before measuring**.
- **Indexing "just in case"** without evidence.
- **Adding `LIMIT 100` to test** and forgetting it.
- **Comparing perf in dev** (small data) and assuming prod behaves the
  same.
- **Forgetting cache effects**: first run slow (disk), next fast (cache).

## Practical exercise

<Exercise
  id="sql-query-opt-exercise"
  title="Mock query plan diagnosis"
  description="Given a plan-like object, return an array of recommendation strings: 'Seq Scan' on table with > 100k rows → 'missing index'; 'Sort' as topOp → 'consider indexed ORDER BY'."
  language="javascript"
  starterCode={`function diagnose(plan) {
  return [];
}

console.log(diagnose({
  topOp: 'Sort',
  scans: [
    { type: 'Seq', table: 'orders', rows: 1000000 },
    { type: 'Index', table: 'users', rows: 5000 },
  ],
}));`}
  solution={`function diagnose(plan) {
  const out = [];
  if (plan.topOp === 'Sort') out.push('consider indexed ORDER BY');
  for (const s of plan.scans || []) {
    if (s.type === 'Seq' && s.rows >= 100000) {
      out.push(\`missing index on \${s.table}\`);
    }
  }
  return out;
}
console.log(diagnose({
  topOp: 'Sort',
  scans: [
    { type: 'Seq', table: 'orders', rows: 1000000 },
    { type: 'Index', table: 'users', rows: 5000 },
  ],
}));`}
  hints={[
    'Check plan.topOp for Sort signal.',
    'Walk plan.scans looking for big Seq Scans.',
  ]}
  tests={[
    {
      description: 'Big Seq Scan detected',
      assert: 'diagnose({topOp:"Hash",scans:[{type:"Seq",table:"orders",rows:200000}]}).join(",").includes("orders")',
    },
    {
      description: 'Sort detected',
      assert: 'diagnose({topOp:"Sort",scans:[]}).join(",").includes("ORDER BY")',
    },
    {
      description: 'healthy plan: no recs',
      assert: 'diagnose({topOp:"Hash",scans:[{type:"Index",table:"users",rows:1000000}]}).length === 0',
    },
  ]}
/>
