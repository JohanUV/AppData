## CASE WHEN and conditional logic

`CASE WHEN` is SQL's if/else. It lets you generate condition-dependent
columns, categorize data, handle NULLs elegantly and move logic from app
to warehouse.

### Two syntaxes

**1. Searched CASE (most used):**

```sql
SELECT
  user_id,
  amount,
  CASE
    WHEN amount < 10 THEN 'small'
    WHEN amount < 100 THEN 'medium'
    WHEN amount < 1000 THEN 'large'
    ELSE 'huge'
  END AS bucket
FROM payments;
```

**2. Simple CASE (when comparing the same thing):**

```sql
SELECT
  status,
  CASE status
    WHEN 'paid' THEN 1
    WHEN 'pending' THEN 0
    ELSE -1
  END AS status_score
FROM orders;
```

`ELSE` is optional but recommended: without it, no match → NULL. The #1
source of "why am I getting unexpected nulls?".

### Typical cases

**Categorization:**

```sql
SELECT
  CASE
    WHEN age < 18 THEN 'minor'
    WHEN age < 30 THEN 'young_adult'
    WHEN age < 60 THEN 'adult'
    ELSE 'senior'
  END AS age_group,
  COUNT(*) AS users
FROM users
GROUP BY 1;
```

**Manual pivots:**

```sql
SELECT
  date_trunc('day', created_at) AS day,
  SUM(CASE WHEN status = 'paid' THEN amount ELSE 0 END) AS revenue_paid,
  SUM(CASE WHEN status = 'refunded' THEN amount ELSE 0 END) AS revenue_refunded,
  SUM(CASE WHEN status = 'pending' THEN amount ELSE 0 END) AS revenue_pending
FROM payments
GROUP BY 1;
```

Classic trick to "pivot" long → wide. Native syntax exists (`PIVOT` in
Snowflake/SQL Server, `crosstab` in Postgres) — but `CASE` works on any
engine.

**Conditional counts:**

```sql
SELECT
  user_id,
  COUNT(*) AS total_orders,
  COUNT(CASE WHEN status = 'paid' THEN 1 END) AS paid_orders,
  COUNT(CASE WHEN amount > 100 THEN 1 END) AS high_value_orders
FROM payments
GROUP BY user_id;
```

`COUNT(CASE WHEN ... THEN 1 END)` counts only matching rows — NULLs
(when condition is false) don't count. In modern Postgres,
`COUNT(*) FILTER (WHERE ...)` is cleaner.

### CASE as NULL handler

```sql
-- Replace NULL with default
SELECT user_id,
  CASE WHEN country IS NULL THEN 'unknown' ELSE country END AS country
FROM users;

-- Equivalent shortcut:
SELECT user_id, COALESCE(country, 'unknown') AS country FROM users;
```

`COALESCE(a, b, c)` returns the first non-null. Prefer it for "if null
then X"; use `CASE` for complex logic.

### IIF, IF, NULLIF

```sql
-- IIF (SQL Server, BigQuery): binary if-else
IIF(amount > 100, 'big', 'small')

-- NULLIF: convert specific value to NULL
NULLIF(division_total, 0)

-- COALESCE: first non-null
COALESCE(nickname, name, 'anon')
```

`NULLIF(x, y)` returns NULL if x = y, else x. Useful to sanitize:

```sql
SELECT amount / NULLIF(quantity, 0) AS unit_price FROM order_items;
-- When quantity = 0, divisor is NULL → result NULL instead of error
```

### Advanced patterns

**Tag first/last/only in partition:**

```sql
WITH ranked AS (
  SELECT
    *,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at) AS rn,
    COUNT(*) OVER (PARTITION BY user_id) AS total
  FROM purchases
)
SELECT
  user_id,
  amount,
  CASE
    WHEN total = 1 THEN 'only'
    WHEN rn = 1 THEN 'first'
    WHEN rn = total THEN 'last'
    ELSE 'middle'
  END AS purchase_position
FROM ranked;
```

### Common mistakes

- **Forgetting `ELSE`**: NULL when no match.
- **Mixing types in THENs**: strict engines fail or coerce.
- **`CASE` when `COALESCE` or `NULLIF` suffice** — less readable.
- **5-level nested CASE WHEN**: factor out into CTE or function.
