## Índices y rendimiento

Tu query funciona en dev con 1.000 filas. Va a producción y muere con
10 millones. La razón casi siempre es la misma: **falta de índice** o
índice mal usado. Esta lección cubre qué son los índices, cuándo
crearlos, cuándo NO, y cómo leer un plan de ejecución para diagnosticar.

### Qué es un índice

Un índice es una **estructura de datos auxiliar** que el motor mantiene
para acelerar lookups. La analogía clásica es el índice alfabético al
final de un libro: te lleva directo a la página sin leer el libro entero.

Sin índice, una query como:

```sql
SELECT * FROM users WHERE email = 'ana@x.com';
```

hace **full table scan**: lee todas las filas y compara email con cada
una. Con 10M filas, son 10M comparaciones.

Con índice en `email`:

```sql
CREATE INDEX idx_users_email ON users(email);
```

el motor consulta una estructura B-tree y encuentra la fila en
~log(10M) ≈ 24 comparaciones. Es la diferencia entre 5 segundos y 0.5
milisegundos.

### Tipos de índices más comunes

- **B-tree** (default casi siempre). Equality, range queries (`<`, `>`,
  `BETWEEN`), `ORDER BY`, `LIKE 'foo%'`.
- **Hash**. Sólo equality, muy rápido. Postgres y otros motores: caso de
  uso nicho.
- **GIN / GiST** (Postgres). Búsqueda full-text, JSON, arrays.
- **Bitmap** (Oracle, columnar). Cardinalidad baja (gender, status).
- **Columnstore** (BigQuery, Redshift, Snowflake). Sin índices
  tradicionales — los warehouses columnares optimizan automáticamente
  por columna.

En OLTP (Postgres, MySQL) trabajas con B-tree. En analítica (BigQuery,
Snowflake) los conceptos cambian: hablas de **clustering** y
**particionamiento** (próxima lección).

### Cuándo crear un índice

<Diagram type="mermaid">
{`flowchart TD
  A[Query lenta] --> B{¿Filtra/joinea<br/>por una columna?}
  B -->|Sí| C{¿Esa columna<br/>tiene alta cardinalidad?}
  C -->|Sí| D[Índice probablemente útil]
  C -->|No| E[Considerar bitmap o saltarse]
  B -->|No| F[Otro problema:<br/>full scan necesario]`}
</Diagram>

**Buenos candidatos:**

- Foreign keys (`user_id`, `order_id`).
- Columnas muy filtradas (`email`, `created_at`).
- Columnas usadas en `ORDER BY` con `LIMIT`.
- Columnas únicas (`UNIQUE INDEX`).

**Malos candidatos:**

- Columnas con muy baja cardinalidad (`is_active`, `gender`) — el motor
  prefiere full scan.
- Columnas muy modificadas: cada UPDATE/INSERT actualiza el índice (costo
  de escritura).
- Columnas casi nunca filtradas.

### Índices compuestos

Cuando filtras por varias columnas:

```sql
-- Para queries con WHERE country = X AND created_at > Y
CREATE INDEX idx_users_country_created
ON users(country, created_at);
```

**Orden importa.** Esta query lo usa:

```sql
WHERE country = 'CO' AND created_at > '2026-01-01';
```

Estas también:

```sql
WHERE country = 'CO';                                    -- usa el prefijo
WHERE country = 'CO' ORDER BY created_at DESC;           -- prefijo + sort
```

Esta NO lo usa:

```sql
WHERE created_at > '2026-01-01';   -- no empieza por country
```

Regla: el índice compuesto se usa de izquierda a derecha. Pon primero la
columna que filtras más, segunda la que filtras siempre que filtras la
primera.

### El costo de los índices

No es gratis tener índices:

- **Espacio**: cada índice ocupa disco (10-30% del tamaño de la tabla).
- **Escritura**: cada INSERT/UPDATE/DELETE actualiza todos los índices
  relevantes. En tablas write-heavy, demasiados índices ralentizan todo.
- **Mantenimiento**: VACUUM, REINDEX en Postgres.

Regla general: **menos es más**. Empieza con índices en FKs y columnas
únicas. Agrega más sólo si EXPLAIN muestra Seq Scan en queries lentas
reales.

### Cosas que invalidan un índice

```sql
-- ❌ Función sobre la columna
WHERE LOWER(email) = 'ana@x.com'
-- ✓ Crea functional index o normaliza al guardar
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- ❌ Wildcard al inicio
WHERE email LIKE '%@gmail.com'
-- ✓ Wildcard al final
WHERE email LIKE 'ana@%'

-- ❌ Cast implícito
WHERE user_id = '123'   -- user_id es INTEGER
-- ✓ Tipo correcto
WHERE user_id = 123

-- ❌ OR a veces fuerza scan
WHERE email = 'a' OR phone = 'b'
-- ✓ UNION de dos queries con índices respectivos
```

### Errores comunes

- **Crear índices por intuición sin ver EXPLAIN**.
- **Sobre-indexar tablas write-heavy**.
- **Olvidar índice en foreign keys**: queries que joinean por FK son
  lentas.
- **Usar funciones sobre la columna en WHERE** sin functional index.
- **Confundir UNIQUE INDEX con CONSTRAINT**: el primero acelera, el
  segundo restringe; en muchos motores son lo mismo internamente.
- **Aplicar conceptos OLTP a warehouse columnar**: BigQuery no usa
  B-trees; ahí lo tuyo es particionamiento + clustering.

## Ejercicio práctico

<Exercise
  id="sql-indexes-exercise"
  title="Detector de queries que invalidan índices"
  description="Dado un WHERE clause como string, devuelve true si parece que invalida un índice convencional (función sobre columna, LIKE con wildcard al inicio)."
  language="javascript"
  starterCode={`function invalidatesIndex(whereClause) {
  return false;
}

console.log(invalidatesIndex("WHERE LOWER(email) = 'a'"));     // true
console.log(invalidatesIndex("WHERE email = 'a'"));             // false
console.log(invalidatesIndex("WHERE name LIKE '%foo%'"));       // true
console.log(invalidatesIndex("WHERE name LIKE 'foo%'"));        // false`}
  solution={`function invalidatesIndex(whereClause) {
  // Detecta funciones tipo LOWER(col), UPPER(col), DATE(col), etc.
  if (/[A-Z]+\\([a-z_]+\\)\\s*=/i.test(whereClause)) return true;
  // Detecta LIKE '%foo' o LIKE '%foo%'
  if (/LIKE\\s+'%/i.test(whereClause)) return true;
  return false;
}
console.log(invalidatesIndex("WHERE LOWER(email) = 'a'"));
console.log(invalidatesIndex("WHERE email = 'a'"));
console.log(invalidatesIndex("WHERE name LIKE '%foo%'"));
console.log(invalidatesIndex("WHERE name LIKE 'foo%'"));`}
  hints={[
    'Regex /[A-Z]+\\([a-z_]+\\)/ detecta funciones aplicadas a columnas.',
    'LIKE seguido de % al inicio del patrón es un anti-patrón clásico.',
  ]}
  tests={[
    {
      description: 'función sobre columna',
      assert: 'invalidatesIndex("WHERE LOWER(email) = \\\'a\\\'") === true',
    },
    {
      description: 'igualdad simple',
      assert: 'invalidatesIndex("WHERE email = \\\'a\\\'") === false',
    },
    {
      description: 'LIKE con wildcard al inicio',
      assert: 'invalidatesIndex("WHERE name LIKE \\\'%foo%\\\'") === true',
    },
    {
      description: 'LIKE con wildcard al final OK',
      assert: 'invalidatesIndex("WHERE name LIKE \\\'foo%\\\'") === false',
    },
  ]}
/>
