## Window functions II: LAG, LEAD, SUM OVER

The window functions from the last lesson give you ranking. Today's let
you **look at other rows** from the current one: previous row (`LAG`),
next (`LEAD`), or all preceding accumulated (`SUM OVER`). Indispensable
for time series analysis.

### LAG and LEAD

{`SELECT
  user_id,
  created_at,
  amount,
  LAG(amount, 1, 0) OVER (PARTITION BY user_id ORDER BY created_at) AS prev_amount,
  LEAD(amount, 1) OVER (PARTITION BY user_id ORDER BY created_at) AS next_amount,

Typical case: detect gaps between events.

```sql
SELECT
  user_id,
  created_at,
  EXTRACT(EPOCH FROM (
    created_at - LAG(created_at) OVER (PARTITION BY user_id ORDER BY created_at)
  )) / 86400 AS days_since_last_order
FROM orders;
```

### Running totals: SUM OVER with ORDER BY

```sql
SELECT
  user_id,
  created_at,
  amount,
  SUM(amount) OVER (PARTITION BY user_id ORDER BY created_at) AS running_total
FROM payments;
```

Each row sees the sum from partition start up to itself. With ORDER BY
inside OVER, SUM becomes cumulative; without, it sums the whole group.

### Frame clauses: fine-grained control

By default, with `ORDER BY`, the frame is "from partition start to
current row". You can change it:

```sql
SUM(amount) OVER (
  PARTITION BY user_id
  ORDER BY created_at
  ROWS BETWEEN 6 PRECEDING AND CURRENT ROW   -- last 7 rows
)
```

Frame types:

- `ROWS BETWEEN N PRECEDING AND M FOLLOWING` — physical row count.
- `RANGE BETWEEN ...` — based on ORDER BY values (handy with dates).
- `UNBOUNDED PRECEDING` — from the start.
- `UNBOUNDED FOLLOWING` — to the end.
- `CURRENT ROW` — the current row.

### 7-day rolling average

```sql
SELECT
  date,
  daily_revenue,
  AVG(daily_revenue) OVER (
    ORDER BY date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS rolling_avg_7d
FROM daily_kpis;
```

`6 PRECEDING + CURRENT ROW` = 7-row window. For exact dates (tolerating
missing days), use `RANGE`:

```sql
RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW
```

### FIRST_VALUE and LAST_VALUE

```sql
SELECT
  user_id,
  created_at,
  amount,
  FIRST_VALUE(amount) OVER (PARTITION BY user_id ORDER BY created_at) AS first_purchase,
  LAST_VALUE(amount) OVER (
    PARTITION BY user_id
    ORDER BY created_at
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS last_purchase
FROM payments;
```

<Callout type="warning" title="LAST_VALUE trap">
  Default frame with ORDER BY ends at CURRENT ROW, so LAST_VALUE = current
  row. To get the actual last of the partition, expand the frame with ROWS
  BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING.
</Callout>

### Case: identify each user's first purchase

```sql
WITH first_purchases AS (
  SELECT
    user_id,
    amount,
    created_at,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at) AS rn
  FROM payments
)
SELECT user_id, amount, created_at
FROM first_purchases
WHERE rn = 1;
```

### Case: day-over-day retention

```sql
WITH daily AS (
  SELECT
    date_trunc('day', event_at) AS day,
    COUNT(DISTINCT user_id) AS dau
  FROM events GROUP BY 1
)
SELECT
  day,
  dau,
  LAG(dau) OVER (ORDER BY day) AS dau_yesterday,
  100.0 * (dau - LAG(dau) OVER (ORDER BY day)) / LAG(dau) OVER (ORDER BY day) AS pct_change
FROM daily
ORDER BY day;
```

### Common mistakes

- **LAST_VALUE without expanded frame**. Returns current row, not last.
- **Forgetting ORDER BY in OVER when needed** — LAG, LEAD, running sums
  become non-deterministic.
- **Confusing ROWS and RANGE**. ROWS counts physical rows; RANGE counts
  values. With irregular spacing it matters.
- **Thinking LAG with NULL "skips"**. It doesn't — it returns the value
  exactly n positions back.

<Exercise

  title="Running total"
  description="Implement `runningTotal(rows, key)` taking an array of objects with `key` and returning rows with `total` = cumulative sum."

  return rows;

  solution={`function runningTotal(rows, key) {
  let acc = 0;
  return rows.map(r => {
    acc += r[key];
    return { ...r, total: acc };
  });
}
console.log(runningTotal([{v:10},{v:20},{v:5}], 'v'));`}
