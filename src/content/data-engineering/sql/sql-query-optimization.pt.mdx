## Otimização de queries

Uma query lenta nunca é mágica. Há um gargalo concreto: full scan, falta
de índice, JOIN explodindo linhas, ou planner estimando mal.

### Processo de diagnóstico

<Diagram type="mermaid">
{`flowchart TD
  A[Query lenta] --> B[Reproduza<br/>com EXPLAIN ANALYZE]
  B --> C{Onde está<br/>o tempo?}
  C -->|Seq Scan| D[Falta índice<br/>ou filtro não-seletivo]
  C -->|Sort| E[ORDER BY caro<br/>+ falta índice]
  C -->|Hash Join enorme| F[Tabela intermediária<br/>infla — revise JOINs]
  C -->|Nested Loop com N alto| G[Subquery correlacionada<br/>ou lookup repetido]
  D --> H[Crie índice apropriado]
  E --> H
  F --> I[Filtre antes,<br/>JOIN pequeno primeiro]
  G --> J[Reescreva como JOIN]`}
</Diagram>

### Leia EXPLAIN como detetive

Comece pela operação mais cara. De dentro para fora.

### Estratégia 1: filtre cedo

```sql
-- ❌ Lento
SELECT *
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.country = 'CO' AND o.status = 'paid';

-- ✓ Melhor
WITH co_users AS (
  SELECT * FROM users WHERE country = 'CO'
),
paid_orders AS (
  SELECT * FROM orders WHERE status = 'paid'
)
SELECT * FROM co_users u LEFT JOIN paid_orders o ON u.id = o.user_id;
```

### Estratégia 2: índices certos

WHERE, JOIN ON, ORDER BY com LIMIT são candidatos.

### Estratégia 3: cuidado com SELECT *

Com índice cobrindo, o motor faz **index-only scan**: nem toca a tabela.

### Estratégia 4: LIMIT muda o plano

Otimizador é mais agressivo com LIMIT pequeno.

### Estratégia 5: agregações caras

`COUNT(DISTINCT col)` em tabelas enormes é caro. Use
`APPROX_COUNT_DISTINCT` (BigQuery, Redshift) — menos de 1% de erro, muito mais
rápido.

### Estratégia 6: particionamento (warehouses)

BigQuery/Snowflake/Redshift: particionamento por data + clustering por
colunas frequentes em WHERE.

### Estratégia 7: evite cartesianos

```sql
SELECT * FROM orders, order_items;   -- cross join implícito!
```

Sempre `JOIN ... ON` explícito.

### Estratégia 8: reescreva subqueries correlacionadas

```sql
-- ❌ N evaluations
SELECT u.name,
  (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS n
FROM users u;

-- ✓ Uma passada
SELECT u.name, COALESCE(o.n, 0) AS n
FROM users u
LEFT JOIN (
  SELECT user_id, COUNT(*) AS n FROM orders GROUP BY user_id
) o ON u.id = o.user_id;
```

### Estratégia 9: estatísticas atualizadas

Postgres usa `ANALYZE`. Programe `VACUUM ANALYZE`.

### Erros comuns

- **Otimizar antes de medir**.
- **Criar índices "por garantia"**.
- **Comparar perf em dev vs prod**.
- **Esquecer efeitos de cache**.

## Exercício prático

<Exercise
  id="sql-query-opt-exercise"
  title="Análise simulada de plano"
  description="Implemente `diagnose(plan)` retornando recomendações."
  language="javascript"
  starterCode={`function diagnose(plan) {
  return [];
}

console.log(diagnose({
  topOp: 'Sort',
  scans: [
    { type: 'Seq', table: 'orders', rows: 1000000 },
    { type: 'Index', table: 'users', rows: 5000 },
  ],
}));`}
  solution={`function diagnose(plan) {
  const out = [];
  if (plan.topOp === 'Sort') out.push('considerar ORDER BY indexado');
  for (const s of plan.scans || []) {
    if (s.type === 'Seq' && s.rows >= 100000) {
      out.push(\`falta índice em \${s.table}\`);
    }
  }
  return out;
}
console.log(diagnose({
  topOp: 'Sort',
  scans: [
    { type: 'Seq', table: 'orders', rows: 1000000 },
    { type: 'Index', table: 'users', rows: 5000 },
  ],
}));`}
  hints={[
    'Verifique plan.topOp.',
    'Percorra plan.scans buscando Seq Scans grandes.',
  ]}
  tests={[
    {
      description: 'Seq Scan grande detectado',
      assert: 'diagnose({topOp:"Hash",scans:[{type:"Seq",table:"orders",rows:200000}]}).join(",").includes("orders")',
    },
    {
      description: 'Sort detectado',
      assert: 'diagnose({topOp:"Sort",scans:[]}).join(",").includes("ORDER BY")',
    },
    {
      description: 'plano saudável',
      assert: 'diagnose({topOp:"Hash",scans:[{type:"Index",table:"users",rows:1000000}]}).length === 0',
    },
  ]}
/>
