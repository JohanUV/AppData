## Indexes and performance

Your query works in dev with 1,000 rows. It hits production and dies on
10 million. The reason is almost always the same: **missing index** or
poorly used one. This lesson covers what indexes are, when to create them,
when NOT to, and how to read an execution plan.

### What an index is

An index is an **auxiliary data structure** the engine maintains to speed
up lookups. Classic analogy: the alphabetical index at the end of a book
takes you to the page without reading the whole thing.

Without an index, a query like:

```sql
SELECT * FROM users WHERE email = 'ana@x.com';
```

does a **full table scan**: reads all rows and compares email. With 10M
rows, 10M comparisons.

With an index on `email`:

```sql
CREATE INDEX idx_users_email ON users(email);
```

the engine queries a B-tree and finds the row in ~log(10M) ≈ 24
comparisons. Difference between 5 seconds and 0.5 ms.

### Common index types

- **B-tree** (default). Equality, range queries, `ORDER BY`, `LIKE
  'foo%'`.
- **Hash**. Equality only, very fast. Niche.
- **GIN / GiST** (Postgres). Full-text search, JSON, arrays.
- **Bitmap** (Oracle, columnar). Low cardinality.
- **Columnstore** (BigQuery, Redshift, Snowflake). No traditional
  indexes — columnar warehouses optimize per column automatically.

In OLTP (Postgres, MySQL) you work with B-trees. In analytics (BigQuery,
Snowflake) concepts shift to **clustering** and **partitioning**.

### When to create an index

<Diagram type="mermaid">
{`flowchart TD
  A[Slow query] --> B{Filters/joins<br/>on a column?}
  B -->|Yes| C{High cardinality?}
  C -->|Yes| D[Index probably useful]
  C -->|No| E[Bitmap or skip]
  B -->|No| F[Different problem:<br/>full scan needed]`}
</Diagram>

**Good candidates:**

- Foreign keys (`user_id`, `order_id`).
- Highly filtered columns (`email`, `created_at`).
- Columns in `ORDER BY` with `LIMIT`.
- Unique columns (`UNIQUE INDEX`).

**Bad candidates:**

- Very low cardinality (`is_active`, `gender`).
- Heavily updated columns: every UPDATE/INSERT updates the index.
- Rarely filtered columns.

### Compound indexes

```sql
-- For queries with WHERE country = X AND created_at > Y
CREATE INDEX idx_users_country_created
ON users(country, created_at);
```

**Order matters.** Used by:

```sql
WHERE country = 'CO' AND created_at > '2026-01-01';
WHERE country = 'CO';                                    -- prefix
WHERE country = 'CO' ORDER BY created_at DESC;           -- prefix + sort
```

NOT used by:

```sql
WHERE created_at > '2026-01-01';   -- doesn't start with country
```

Rule: compound index is used left-to-right.

### Index cost

Indexes aren't free:

- **Disk space**: each one takes 10-30% of the table size.
- **Writes**: every INSERT/UPDATE/DELETE updates relevant indexes.
- **Maintenance**: VACUUM, REINDEX in Postgres.

Rule of thumb: **less is more**. Start with FK + unique columns. Add more
only if EXPLAIN shows Seq Scan on real slow queries.

### Things that invalidate an index

```sql
-- ❌ Function over column
WHERE LOWER(email) = 'ana@x.com'
-- ✓ Functional index or normalize on save
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- ❌ Leading wildcard
WHERE email LIKE '%@gmail.com'
-- ✓ Trailing wildcard
WHERE email LIKE 'ana@%'

-- ❌ Implicit cast
WHERE user_id = '123'   -- user_id is INTEGER
-- ✓ Right type
WHERE user_id = 123

-- ❌ OR sometimes forces scan
WHERE email = 'a' OR phone = 'b'
-- ✓ UNION of two queries with respective indexes
```

### Common mistakes

- **Indexing on hunches without EXPLAIN**.
- **Over-indexing write-heavy tables**.
- **Forgetting index on FKs**: JOINs by FK become slow.
- **Using functions over columns** without functional index.
- **Confusing UNIQUE INDEX with CONSTRAINT**: many engines use the same.
- **Applying OLTP concepts to columnar warehouses**: BigQuery doesn't
  use B-trees; partition + cluster.

<Exercise

  description="Given a WHERE clause as string, return true if it likely invalidates a regular index (function over column, LIKE with leading wildcard)."

  starterCode={`function invalidatesIndex(whereClause) {
  return false;

console.log(invalidatesIndex("WHERE LOWER(email) = 'a'"));

console.log(invalidatesIndex("WHERE name LIKE 'foo%'"));`}
  solution={`function invalidatesIndex(whereClause) {
  if (/[A-Z]+\\([a-z_]+\\)\\s*=/i.test(whereClause)) return true;
  if (/LIKE\\s+'%/i.test(whereClause)) return true;
  return false;
}
console.log(invalidatesIndex("WHERE LOWER(email) = 'a'"));
console.log(invalidatesIndex("WHERE email = 'a'"));
console.log(invalidatesIndex("WHERE name LIKE '%foo%'"));
console.log(invalidatesIndex("WHERE name LIKE 'foo%'"));`}
  hints={[
    'Regex /[A-Z]+\\([a-z_]+\\)/ detects functions on columns.',
    'LIKE followed by % at start is a classic anti-pattern.',
  ]}
  tests={[
    {
      description: 'function over column',
      assert: 'invalidatesIndex("WHERE LOWER(email) = \\\'a\\\'") === true',
    },
    {
      description: 'simple equality',
      assert: 'invalidatesIndex("WHERE email = \\\'a\\\'") === false',
    },
    {
      description: 'LIKE with leading wildcard',
      assert: 'invalidatesIndex("WHERE name LIKE \\\'%foo%\\\'") === true',
    },
    {
      description: 'LIKE with trailing wildcard OK',
